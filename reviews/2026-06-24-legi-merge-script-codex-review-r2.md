# Code Review: `/tmp/juri-merge.sh` r2

## Findings

### BLOCKER: The "decision-ish" smoke can pass without any jurisprudence result

Step 7 is meant to prove both corpora are searchable through the merged index, but the second smoke command uses `search --kind all --mode dense` and only counts the total `candidates` array (`/tmp/juri-merge.sh:179-184`). That does not enforce a decision result. A dense all-corpus query can return only LEGI article candidates and still set `dec_n >= 1`, so the script can print `MERGE + VALIDATION COMPLETE` even if jurisprudence retrieval is broken.

This matters because the CLI does not currently accept `--kind decision`: `search_payload` rejects `LegalKind::Decision` and allows only `--kind all` or `--kind code` (`crates/jurisearch-cli/src/main.rs:2473-2477`). The storage layer will apply decision-only predicates when a decision filter is present, because any non-empty `DecisionFilters` predicate adds `AND d.kind = 'decision'` (`crates/jurisearch-storage/src/retrieval.rs:116-124`), and otherwise `--kind all` has no kind predicate (`crates/jurisearch-cli/src/main.rs:2653-2657`; `crates/jurisearch-storage/src/retrieval.rs:276-281`).

Concrete fix: make the smoke assert the returned rows, not just the count. For example, keep `--kind all` but parse the JSON and require at least one candidate with `kind == "decision"` and `source in {"cass","capp","inca","jade"}`. Alternatively, add a broad decision filter that forces `d.kind = 'decision'` (for example a deliberately wide `--decided-from` if that matches the intended corpus), then still assert the candidate kind/source in Python so the script fails loudly if CLI semantics change. The `code_n` smoke should similarly assert every or at least one returned candidate has `kind == "article"` / `source == "legi"` rather than relying only on `--kind code`.
