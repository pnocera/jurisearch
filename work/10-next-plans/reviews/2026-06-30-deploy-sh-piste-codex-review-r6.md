## Findings

### BLOCKER: PISTE-triggered env refresh can erase preserved production credentials

[deploy.sh](/home/pierre/Work/jurisearch/deploy.sh:193) stages a complete `producer.env`, but every unset deploy-time credential is written as an empty assignment at [deploy.sh](/home/pierre/Work/jurisearch/deploy.sh:199), [deploy.sh](/home/pierre/Work/jurisearch/deploy.sh:204), [deploy.sh](/home/pierre/Work/jurisearch/deploy.sh:209), and [deploy.sh](/home/pierre/Work/jurisearch/deploy.sh:214). The new broadened write trigger at [deploy.sh](/home/pierre/Work/jurisearch/deploy.sh:369) installs that entire staged file when any one of `OPENROUTER_API_KEY`, `PISTE_API_KEY`, `PISTE_OAUTH_CLIENT_ID`, or `PISTE_OAUTH_CLIENT_SECRET` is non-empty.

That regresses upgrade preservation for the live host. For example, an operator adding PISTE to an already-working deployment may export only the three new `PISTE_*` values and not re-export the already-installed `OPENROUTER_API_KEY`; the new trigger will overwrite the existing remote `producer.env` with `OPENROUTER_API_KEY=` and the deploy still verifies green, but the first real update will fail because embedding reads `OPENROUTER_API_KEY` from the environment. The same failure mode applies within PISTE itself: rotating only one PISTE value blanks the other two, and rotating only `OPENROUTER_API_KEY` later blanks previously installed PISTE credentials.

Fix: do not replace the whole environment file from an incomplete deploy-time snapshot. On the remote side, merge non-empty staged assignments into the existing `producer.env` while preserving existing values for omitted keys, then install the merged file as `root:root 0600`. Alternatively, fail closed on an existing `producer.env` unless the operator provides the full managed credential set being rewritten. Keep the Phase-6 mode/owner assertion after the merge.
