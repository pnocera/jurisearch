# Review - 02-auto-update-server-crons.md (round 3)

## Findings

### BLOCKER - The `core` package lock is scoped too narrowly for two fetch groups that both write the same `core` outbox

The round-2 corpus attribution issue is fixed in the prose, but the revised lock model now lets the two fetch groups overlap through the shared producer DB/outbox. The plan says the orchestration command takes a fetch group and then packages `core` (`work/10-next-plans/02-auto-update-server-crons.md:192-225`), and Phase 3 explicitly says a per-group fetch lock lets the two groups "fetch/ingest independently" while only the publish step takes `state_dir/package-core.lock` (`work/10-next-plans/02-auto-update-server-crons.md:365-372`).

That does not match the current package boundary. In source, all DILA sources map to `core` (`crates/jurisearch-package/src/corpus.rs:75-81`), and both LEGI and jurisprudence projections emit outbox rows by calling `corpus_for_source()` in the same mutation transaction (`crates/jurisearch-storage/src/projection/legi.rs:212-230`, `crates/jurisearch-storage/src/projection/decisions.rs:160-178`). The package builder has no fetch-group/source/run eligibility boundary: `build_incremental()` takes only `corpus`, reads the latest catalog row for that corpus, then asks `scopes_changed_for_corpus_with_client()` for every row where `package_change_log.corpus = $1` in the next change-seq window (`crates/jurisearch-package-build/src/incremental.rs:139-191`, `crates/jurisearch-storage/src/outbox.rs:243-257`). `producer_cycle()` likewise takes only `corpus` and publishes whatever that corpus window contains (`crates/jurisearch-package-build/src/cycle.rs:59-93`).

So a concrete race remains: the jurisprudence timer can ingest CASS/CAPP/INCA/JADE rows into the shared `core` outbox, then pause before enrichment/embedding/publish; meanwhile the legislation timer can finish its own group, acquire the package lock at publish time, and build a `core` incremental that includes the partially processed jurisprudence scopes. The existing per-corpus build lock and outbox fence serialize the build snapshot (`crates/jurisearch-package-build/src/incremental.rs:97-125`, `crates/jurisearch-storage/src/outbox.rs:143-159`), but they do not isolate the caller's preceding ingest/enrich/embed workflow.

Recommended fix: keep per-source/per-group fetch locks for remote download if useful, but introduce one shared `core` update lock before any DB-mutating phase for sources that package as `core`, and hold it through ingest, enrichment, embedding, and `producer_cycle("core")`. If concurrent DB mutation by group is a requirement, that is a larger design: add completed-run/source eligibility to the outbox and teach the package builder to package only completed workflow windows. The current repository does not have that boundary. Add a regression test where a jurisprudence update is held after ingest but before embed; a concurrent legislation timer must not publish a `core` package containing those jurisprudence scopes until the jurisprudence workflow is complete.
