## Verified

The prior publish-resumability blocker is fixed for the operated `producer_cycle` path. The cycle now resumes a pending staged artifact before considering a new build (`crates/jurisearch-package-build/src/cycle.rs:123`), builds the fresh incremental directly into the served-root staging slot (`crates/jurisearch-package-build/src/cycle.rs:132`), publishes it before marking the catalog row `published` (`crates/jurisearch-package-build/src/cycle.rs:152`), and only then rebuilds/publishes the remote manifest (`crates/jurisearch-package-build/src/cycle.rs:160`). The incremental builder writes the embedded manifest into that artifact directory before inserting the `"built"` catalog row (`crates/jurisearch-package-build/src/incremental.rs:500`, `crates/jurisearch-package-build/src/incremental.rs:508`). On retry, `resume_pending` derives the same `package_id` from the staged manifest, republishes that same artifact, marks the row, and clears the slot (`crates/jurisearch-package-build/src/cycle.rs:192`). I do not see a remaining operated-path ordering where the manifest or update checkpoint advances over a missing served artifact.

The manual `jurisearch_package` flow remains compatible with the unchanged catalog/manifest selection model. `build`, `publish`, and `publish-manifest` are still explicit steps (`crates/jurisearch-package-build/src/bin/jurisearch_package.rs:175`, `crates/jurisearch-package-build/src/bin/jurisearch_package.rs:241`, `crates/jurisearch-package-build/src/bin/jurisearch_package.rs:250`), and `build_remote_manifest` still reads the ordered catalog chain and loudly fails on a missing published artifact (`crates/jurisearch-package-build/src/remote_manifest.rs:93`, `crates/jurisearch-package-build/src/remote_manifest.rs:211`). The existing missing-artifact test still exercises that behavior (`crates/jurisearch-package-build/tests/publish_distribution.rs:299`).

`publish_package` is safe for the resume/retry case. If the destination package directory already exists, it compares the existing and incoming embedded artifact digests and no-ops on a match; a different digest for the same id is rejected (`crates/jurisearch-package-build/src/publish.rs:55`). A retry after a successful copy but before catalog marking therefore converges instead of replacing the live package.

The provision password fix is source-correct. `DatabaseConfig` now includes optional `read_password_file` alongside `writer_password_file` (`crates/jurisearch-producer/src/config.rs:68`, `crates/jurisearch-producer/src/config.rs:74`), validation permission-checks both files (`crates/jurisearch-producer/src/config.rs:264`), and `provision_config()` feeds both loaded secrets into `RoleSpec` (`crates/jurisearch-producer/src/config.rs:350`). The storage provisioner uses those values both for `ALTER ROLE ... PASSWORD` and for writer/read postcondition probes (`crates/jurisearch-storage/src/backend.rs:700`, `crates/jurisearch-storage/src/provision.rs:217`). The new unit tests cover writer and optional read password propagation (`crates/jurisearch-producer/tests/config_and_fingerprint.rs:47`, `crates/jurisearch-producer/tests/config_and_fingerprint.rs:65`).

The checkpoint high-water mark now carries real package coordinates. `ProducerCycleReport` exposes `head_sequence` and `included_change_seq_high` (`crates/jurisearch-package-build/src/cycle.rs:63`), `published_head` reads them from the newest catalog row after the manifest publish (`crates/jurisearch-package-build/src/cycle.rs:221`), and `run_update` copies them into `PackageHighWaterMark` (`crates/jurisearch-producer/src/update.rs:182`). For an empty outbox, the report reflects the current catalog head rather than all `None`, which is the sensible checkpoint value for a successful no-op cycle.

The prior non-findings still hold. The producer update path is in-process and uses `ProducerConfig::writer_handle()` / `WriterHandle`, with `ManagedPostgres` only in tests (`crates/jurisearch-producer/src/update.rs:147`, `crates/jurisearch-producer/src/config.rs:367`). Fetch runs before the `update-core` lock, while ingest, enrich, embed, and `producer_cycle` run under the same lock (`crates/jurisearch-producer/src/update.rs:116`, `crates/jurisearch-producer/src/update.rs:146`). Archive ingest selection still keys on archive/journal state, not package `change_seq` (`crates/jurisearch-producer/src/update.rs:227`). The provider `request_model` remains separate from the storage fingerprint in both config and embedding requests (`crates/jurisearch-producer/src/config.rs:296`, `crates/jurisearch-producer/src/update.rs:321`), and the v1 single-corpus invariant remains enforced as `core` (`crates/jurisearch-producer/src/config.rs:228`).

The new resumability test is not false-green for the original failure mode: it asserts the row is `"built"` while the served package is absent and staging exists, the manifest head stays at the baseline before retry, retry publishes the same `core-1-2` package, the row becomes `"published"`, and a later no-op leaves only one incremental catalog row (`crates/jurisearch-producer/tests/update_publish_gated.rs:327`, `crates/jurisearch-producer/tests/update_publish_gated.rs:353`, `crates/jurisearch-producer/tests/update_publish_gated.rs:385`). The checkpoint assertion checks non-`None` real values after a one-delta cycle (`crates/jurisearch-producer/tests/update_publish_gated.rs:242`).
