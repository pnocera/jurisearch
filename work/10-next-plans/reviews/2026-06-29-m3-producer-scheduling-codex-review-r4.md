# M3 Producer Scheduling Codex Review r4

## Findings

No BLOCKER/WARN/NIT findings.

## Notes

The r4 discard-and-rebuild rebaseline design is implemented as described. `rebaseline_cycle_faulted` inspects `.staging/pending` before building, discards staged `PackageKind::Rebaseline` artifacts by deleting only a matching unpublished catalog row and clearing the slot, resumes staged incrementals, then always calls `build_rebaseline` for a fresh rebaseline from the current DB state (`crates/jurisearch-package-build/src/cycle.rs:282`, `crates/jurisearch-package-build/src/cycle.rs:284`, `crates/jurisearch-package-build/src/cycle.rs:288`, `crates/jurisearch-package-build/src/cycle.rs:297`, `crates/jurisearch-package-build/src/cycle.rs:299`). I do not see a remaining path that resumes/publishes a stale staged rebaseline or then adopts a baseline that was not represented by the newly built package.

`delete_unpublished_package_row` is guarded by `status = 'built'`, so it cannot delete a served `published` row (`crates/jurisearch-storage/src/package_catalog.rs:210`, `crates/jurisearch-storage/src/package_catalog.rs:217`). The catalog and manifest selection semantics are otherwise unchanged: `latest_package_for_corpus` and `catalog_rows_for_corpus` still read the catalog without status filtering (`crates/jurisearch-storage/src/package_catalog.rs:155`, `crates/jurisearch-storage/src/package_catalog.rs:249`), and `build_remote_manifest` still builds from catalog rows plus published artifacts (`crates/jurisearch-package-build/src/remote_manifest.rs:93`, `crates/jurisearch-package-build/src/remote_manifest.rs:100`, `crates/jurisearch-package-build/src/remote_manifest.rs:133`). That preserves the manual `jurisearch_package` flow while the rebaseline discard path removes the specific orphaned `built` row that would otherwise pollute chain-head selection.

Per-source adoption no longer collapses the group to a max timestamp. The producer recomputes routing under the single `update-core` lock (`crates/jurisearch-producer/src/update.rs:219`, `crates/jurisearch-producer/src/update.rs:232`), publishes the fresh rebaseline under that lock, and only then writes each `(source, baseline)` marker from `new_baselines` individually (`crates/jurisearch-producer/src/update.rs:277`, `crates/jurisearch-producer/src/update.rs:294`, `crates/jurisearch-producer/src/update.rs:574`, `crates/jurisearch-producer/src/update.rs:579`, `crates/jurisearch-producer/src/update.rs:581`). The multi-source unit test covers distinct `cass`/`inca` baselines and verifies both markers are written to their own per-source files (`crates/jurisearch-producer/src/update.rs:693`, `crates/jurisearch-producer/src/update.rs:703`, `crates/jurisearch-producer/src/update.rs:711`, `crates/jurisearch-producer/src/update.rs:716`).

The uncataloged staging phantom window is closed for the incremental resume path. `resume_pending` now reads the staged identity, requires a matching `package_catalog` row before publishing, discards uncataloged staging, and `mark_package_published` errors unless exactly one row is updated (`crates/jurisearch-package-build/src/cycle.rs:383`, `crates/jurisearch-package-build/src/cycle.rs:388`, `crates/jurisearch-package-build/src/cycle.rs:393`, `crates/jurisearch-package-build/src/cycle.rs:408`, `crates/jurisearch-package-build/src/cycle.rs:412`). A cataloged staged incremental still resumes the same package exactly once, and the new uncataloged-discard test covers the phantom case (`crates/jurisearch-producer/tests/update_publish_gated.rs:353`, `crates/jurisearch-producer/tests/update_publish_gated.rs:362`, `crates/jurisearch-producer/tests/update_publish_gated.rs:479`, `crates/jurisearch-producer/tests/update_publish_gated.rs:484`, `crates/jurisearch-producer/tests/update_publish_gated.rs:497`).

The author-noted residual window is fail-stop rather than client-visible corruption: if rebaseline `publish_package` succeeds but `mark_package_published` fails, the manifest refresh has not run yet (`crates/jurisearch-package-build/src/cycle.rs:312`, `crates/jurisearch-package-build/src/cycle.rs:313`, `crates/jurisearch-package-build/src/cycle.rs:325`, `crates/jurisearch-package-build/src/cycle.rs:332`). On retry, the prior `built` row is discarded and a fresh artifact is built; if the same deterministic package id already exists with different content, `publish_package` rejects the mismatch (`crates/jurisearch-package-build/src/publish.rs:56`, `crates/jurisearch-package-build/src/publish.rs:62`). Because the served manifest is only published after the row is marked, clients cannot have discovered the unreferenced artifact. I consider this acceptable for v1.

The new gated tests cover the r4 invariants that previously failed: same-id discard/rebuild after a pre-publish rebaseline failure, stale B1 rebaseline discarded and rebuilt for current B2, per-source adoption without collapse, and uncataloged staged artifact discard (`crates/jurisearch-package-build/tests/rebaseline_cycle_loopback.rs:222`, `crates/jurisearch-package-build/tests/rebaseline_cycle_loopback.rs:333`, `crates/jurisearch-package-build/tests/rebaseline_cycle_loopback.rs:373`, `crates/jurisearch-package-build/tests/rebaseline_cycle_loopback.rs:474`, `crates/jurisearch-producer/src/update.rs:693`, `crates/jurisearch-producer/tests/update_publish_gated.rs:413`). I did not run the test suite because the instruction was not to modify any other files, and a cargo test run would write build artifacts.

The previously accepted non-rebaseline fixes still hold on source inspection: routing and the incremental backstop are under the update lock (`crates/jurisearch-producer/src/update.rs:219`, `crates/jurisearch-producer/src/update.rs:312`), systemd/data paths are validated as absolute (`crates/jurisearch-producer/src/config.rs:375`, `crates/jurisearch-producer/src/config.rs:397`), run IDs include subsecond time and pid (`crates/jurisearch-producer/src/update.rs:545`, `crates/jurisearch-producer/src/update.rs:563`), and status includes stale-by-age in the overall stale signal with a non-masked regression test (`crates/jurisearch-producer/src/status.rs:173`, `crates/jurisearch-producer/src/status.rs:198`, `crates/jurisearch-producer/tests/run_record_and_status.rs:274`, `crates/jurisearch-producer/tests/run_record_and_status.rs:321`).

VERDICT: GO
