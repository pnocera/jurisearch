### BLOCKER: merge-upsert still is not value-safe for opaque credential values

[deploy.sh](/home/pierre/Work/jurisearch/deploy.sh:368) extracts the staged value as text, but [deploy.sh](/home/pierre/Work/jurisearch/deploy.sh:371) passes that secret through `awk -v val="$v"` before writing it back into `producer.env`. `awk -v` processes C-style escape sequences in assignment values, so a credential containing a literal backslash sequence is silently rewritten. For example, a staged value containing the two bytes `\n` is emitted as an actual newline, `\t` as a tab, and `\b` as a backspace. That violates the r7 requirement that credential values be treated as opaque literals, and the deploy can still verify green while installing a corrupted re-exported key/secret.

Fix: do not pass secret values through `awk -v` or any replacement channel that interprets escapes. Keep the provided `KEY=value` lines as input data and have the merge read values from a file, using only non-secret key names in `-v`. For example, parse `"$STAGING/producer.env"` and `"$merged"` in a single file-driven `awk` program, storing `substr($0, length(k) + 2)` for matched managed keys from the staged file and printing those stored input strings back out. Alternatively, use a small literal-safe parser that writes complete staged lines without re-encoding the value. Keep the final `install -o root -g root -m 0600` and Phase-6 assertion.
