# Codex Review: M1-B S2/S3 migrations + external DB provisioning

## Findings

### BLOCKER: External writer is declared writable but cannot write the public ingest accounting tables

`provision_external_db`'s activation postcondition only proves the writer can upsert `public.index_manifest` ([crates/jurisearch-storage/src/provision.rs:222](/home/pierre/Work/jurisearch-worktrees/m1b-migrations-provision/crates/jurisearch-storage/src/provision.rs:222)). The actual producer write path also writes the public ingest accounting tables: `INSERT/UPDATE ingest_run` ([crates/jurisearch-storage/src/ingest_accounting/runs.rs:60](/home/pierre/Work/jurisearch-worktrees/m1b-migrations-provision/crates/jurisearch-storage/src/ingest_accounting/runs.rs:60), [crates/jurisearch-storage/src/ingest_accounting/runs.rs:128](/home/pierre/Work/jurisearch-worktrees/m1b-migrations-provision/crates/jurisearch-storage/src/ingest_accounting/runs.rs:128)), `INSERT/UPDATE/DELETE ingest_member` ([crates/jurisearch-storage/src/ingest_accounting/members.rs:75](/home/pierre/Work/jurisearch-worktrees/m1b-migrations-provision/crates/jurisearch-storage/src/ingest_accounting/members.rs:75), [crates/jurisearch-storage/src/ingest_accounting/members.rs:113](/home/pierre/Work/jurisearch-worktrees/m1b-migrations-provision/crates/jurisearch-storage/src/ingest_accounting/members.rs:113)), and `INSERT ingest_error` plus `UPDATE ingest_member` ([crates/jurisearch-storage/src/ingest_accounting/errors.rs:41](/home/pierre/Work/jurisearch-worktrees/m1b-migrations-provision/crates/jurisearch-storage/src/ingest_accounting/errors.rs:41), [crates/jurisearch-storage/src/ingest_accounting/errors.rs:59](/home/pierre/Work/jurisearch-worktrees/m1b-migrations-provision/crates/jurisearch-storage/src/ingest_accounting/errors.rs:59)). Those tables are created by migration 3 in `public`, including `bigserial` sequences on `ingest_member` and `ingest_error` ([crates/jurisearch-storage/src/migrations.rs:139](/home/pierre/Work/jurisearch-worktrees/m1b-migrations-provision/crates/jurisearch-storage/src/migrations.rs:139), [crates/jurisearch-storage/src/migrations.rs:155](/home/pierre/Work/jurisearch-worktrees/m1b-migrations-provision/crates/jurisearch-storage/src/migrations.rs:155), [crates/jurisearch-storage/src/migrations.rs:178](/home/pierre/Work/jurisearch-worktrees/m1b-migrations-provision/crates/jurisearch-storage/src/migrations.rs:178)).

`provision_roles` first revokes the writer's entire public table surface ([crates/jurisearch-storage/src/backend.rs:550](/home/pierre/Work/jurisearch-worktrees/m1b-migrations-provision/crates/jurisearch-storage/src/backend.rs:550)), then grants public DML only on `index_manifest`, `schema_migrations`, `package_change_log`, and `package_catalog` ([crates/jurisearch-storage/src/backend.rs:563](/home/pierre/Work/jurisearch-worktrees/m1b-migrations-provision/crates/jurisearch-storage/src/backend.rs:563)). It does not restore DML on `public.ingest_run`, `public.ingest_member`, or `public.ingest_error`, and it does not grant `USAGE` on their sequences. A freshly provisioned external database can therefore return `writer_can_write = true` while the producer fails as soon as it records ingest accounting.

Fix: include the ingest accounting tables in the explicit writer write surface and grant `USAGE, SELECT` on their public sequences, or move them into a shared constant with `NON_GENERATION_TABLES` so the provisioning grants cannot drift from the schema. Strengthen the S3 postcondition/test to run representative ingest accounting writes through the provisioned writer role, not just an `index_manifest` upsert.

### WARN: `run_migrations_on` checks extension privileges before rejecting an unsupported newer schema

`run_migrations_on` calls `ensure_required_extensions` before it creates/reads `schema_migrations` and before it checks `SchemaVersionAhead` ([crates/jurisearch-storage/src/migrations.rs:1213](/home/pierre/Work/jurisearch-worktrees/m1b-migrations-provision/crates/jurisearch-storage/src/migrations.rs:1213), [crates/jurisearch-storage/src/migrations.rs:1236](/home/pierre/Work/jurisearch-worktrees/m1b-migrations-provision/crates/jurisearch-storage/src/migrations.rs:1236)). The old path read the applied versions and rejected `database_version > CURRENT_SCHEMA_VERSION` before applying any migration SQL. With the new ordering, a database that is already ahead of this binary but is missing one required extension can either be mutated by extension creation or fail with `ExtensionPrivilege` instead of the more important `SchemaVersionAhead`.

Fix: create/read `schema_migrations` and perform the version-ahead check before extension preflight. Then run `ensure_required_extensions` only once the database is known to be supported and before applying pending migrations or validating a current schema.

### WARN: Extension preflight changed migration failure atomicity

The new preflight executes `CREATE EXTENSION IF NOT EXISTS ...` in autocommit mode before the per-migration transaction loop ([crates/jurisearch-storage/src/migrations.rs:1178](/home/pierre/Work/jurisearch-worktrees/m1b-migrations-provision/crates/jurisearch-storage/src/migrations.rs:1178), [crates/jurisearch-storage/src/migrations.rs:1249](/home/pierre/Work/jurisearch-worktrees/m1b-migrations-provision/crates/jurisearch-storage/src/migrations.rs:1249)). Historically migration 1's extension DDL ran inside the same `BEGIN ... COMMIT` block as the rest of migration 1 and its version stamp. Now, for example, `vector` can be installed successfully and `pg_search` can then fail with insufficient privilege, leaving a partially changed database with no `schema_migrations` row. That violates the stated "same ordering/transaction boundaries" guarantee even though the version-stamp write itself remains byte-identical.

Fix: perform the required-extension preflight inside the transaction that applies migration 1, while still mapping SQLSTATE `42501` to `StorageError::ExtensionPrivilege`. If the schema is already at/above migration 1 and extensions need repair, handle that as a separate explicit validation/repair step after the schema-version check so unsupported-version rejection still wins.

## Notes

The public S2/S3 APIs are connection-based: `run_migrations_on` takes a borrowed `postgres::Client`, `ManagedPostgres::run_migrations` is a thin delegate, and `ProvisionConfig` contains only `ConnectionConfig` plus role metadata. I did not rerun the PG integration tests because the review instructions state the required extensions are not discoverable in this environment and those tests skip here.
