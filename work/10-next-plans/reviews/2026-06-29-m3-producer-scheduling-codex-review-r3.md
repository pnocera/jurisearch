# M3 Producer Scheduling Codex Review r3

## Findings

BLOCKER: The adoption equality key is not sound for the multi-source `jurisprudence` group, so a resumed package can still adopt a source baseline that was not in the published artifact. `jurisprudence` runs over four sources (`cass`, `inca`, `capp`, `jade`) (`crates/jurisearch-producer/src/config.rs:619`, `crates/jurisearch-producer/src/config.rs:621`), and `group_run_kind` returns every pending `(source, fetched_baseline)` for the group (`crates/jurisearch-producer/src/baseline.rs:163`, `crates/jurisearch-producer/src/baseline.rs:172`, `crates/jurisearch-producer/src/baseline.rs:178`). But the identity used for both build intent and adoption comparison collapses that whole set to only the maximum parsed archive timestamp (`crates/jurisearch-producer/src/update.rs:563`, `crates/jurisearch-producer/src/update.rs:570`, `crates/jurisearch-producer/src/update.rs:571`, `crates/jurisearch-producer/src/update.rs:572`), and `baselines_adopted_by` adopts the entire current `new_baselines` slice whenever that one string matches (`crates/jurisearch-producer/src/update.rs:588`, `crates/jurisearch-producer/src/update.rs:593`, `crates/jurisearch-producer/src/update.rs:594`). Repro shape: run A sees and stages a `cass` rebaseline at timestamp `20250713140000`, then crashes before publish. Before retry, `inca` fetch advances to a newer-than-adopted baseline whose timestamp is `20250712140000` (or the same timestamp), while `cass` is still pending at `20250713140000`. The resumed artifact reports `core-20250713140000`, and the current run derives the same max timestamp from `{cass: 20250713140000, inca: 20250712140000}`, so `baselines_adopted_by` returns both markers. `inca` is then written as adopted at `crates/jurisearch-producer/src/update.rs:295` and `crates/jurisearch-producer/src/update.rs:299` even though the published snapshot was built before that `inca` baseline existed. That silently drops the later `inca` rebaseline. The published/adopted identity needs to encode the per-source baseline set, or adoption needs to compare each source against per-source identity from the published artifact.

BLOCKER: A crash after writing the staged rebaseline manifest but before inserting its catalog row can publish an uncataloged rebaseline and still adopt its baseline marker. `build_media_package` writes the staged artifact manifest before the catalog insert (`crates/jurisearch-package-build/src/baseline.rs:468`, `crates/jurisearch-package-build/src/baseline.rs:471`, `crates/jurisearch-package-build/src/baseline.rs:475`), so that crash window can leave `.staging/pending/manifest.json` with no `package_catalog` row. On the next run, `resume_pending` treats any pending manifest as resumable, publishes it, calls `mark_package_published`, and deletes the staging slot (`crates/jurisearch-package-build/src/cycle.rs:358`, `crates/jurisearch-package-build/src/cycle.rs:371`, `crates/jurisearch-package-build/src/cycle.rs:372`, `crates/jurisearch-package-build/src/cycle.rs:373`). `mark_package_published` does not check that the `UPDATE` matched a row (`crates/jurisearch-package-build/src/cycle.rs:390`, `crates/jurisearch-package-build/src/cycle.rs:391`, `crates/jurisearch-package-build/src/cycle.rs:395`). For a rebaseline, `rebaseline_cycle_faulted` then stops on that resumed artifact and returns its embedded `baseline_id` (`crates/jurisearch-package-build/src/cycle.rs:270`, `crates/jurisearch-package-build/src/cycle.rs:271`, `crates/jurisearch-package-build/src/cycle.rs:273`), but `build_remote_manifest` is catalog-driven (`crates/jurisearch-package-build/src/remote_manifest.rs:93`, `crates/jurisearch-package-build/src/remote_manifest.rs:94`) and will not reference the uncataloged package. The producer can then adopt the baseline marker from the returned report (`crates/jurisearch-producer/src/update.rs:294`, `crates/jurisearch-producer/src/update.rs:299`) even though the served manifest remains on the prior cataloged head. This is still a partial-publish/adoption window; the resume path needs to verify the staged artifact has a matching catalog row before publishing/adopting it, or discard/rebuild uncataloged staging safely.

## Notes

The r2 single-source stale-resume fix is present: `resume_pending` reads the embedded manifest identity and carries the staged artifact's `baseline_id` into `ResumedPackage` (`crates/jurisearch-package-build/src/cycle.rs:362`, `crates/jurisearch-package-build/src/cycle.rs:364`, `crates/jurisearch-package-build/src/cycle.rs:367`, `crates/jurisearch-package-build/src/cycle.rs:374`), and `RebaselineCycleReport` returns that published identity (`crates/jurisearch-package-build/src/cycle.rs:322`, `crates/jurisearch-package-build/src/cycle.rs:325`). In the simple B1/B2 case, the producer no longer adopts the newer current marker when the resumed artifact reports the older id (`crates/jurisearch-producer/src/update.rs:593`).

The previously accepted r2 items still look source-correct: routing is recomputed under the single `update-core` lock before DB mutation (`crates/jurisearch-producer/src/update.rs:219`, `crates/jurisearch-producer/src/update.rs:232`), adoption writes happen under that same locked span after publish (`crates/jurisearch-producer/src/update.rs:277`, `crates/jurisearch-producer/src/update.rs:299`), the ordinary incremental backstop still refuses pending baseline boundaries (`crates/jurisearch-producer/src/update.rs:320`, `crates/jurisearch-producer/src/update.rs:321`), absolute systemd/data/state paths are validated (`crates/jurisearch-producer/src/config.rs:375`, `crates/jurisearch-producer/src/config.rs:397`), run ids include subsecond time and pid (`crates/jurisearch-producer/src/update.rs:163`, `crates/jurisearch-producer/src/update.rs:168`), and the status implementation includes `any_stale` in the overall stale signal (`crates/jurisearch-producer/src/status.rs:198`, `crates/jurisearch-producer/src/status.rs:201`). The stale-by-age test now seeds a recent successful `legislation` run before making only `jurisprudence` old, so the overall assertion is no longer masked by `any_never_ran` (`crates/jurisearch-producer/tests/run_record_and_status.rs:274`, `crates/jurisearch-producer/tests/run_record_and_status.rs:281`, `crates/jurisearch-producer/tests/run_record_and_status.rs:319`, `crates/jurisearch-producer/tests/run_record_and_status.rs:321`).
