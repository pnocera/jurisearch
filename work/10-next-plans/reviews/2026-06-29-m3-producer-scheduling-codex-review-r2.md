# M3 Producer Scheduling Codex Review r2

## Findings

BLOCKER: A resumed staged rebaseline can cause the producer to adopt the wrong, newer upstream baseline. In `rebaseline_cycle_faulted`, any pending package whose embedded kind is `Rebaseline` is published and treated as the whole cycle without comparing the staged artifact's embedded `baseline_id` to the current run's `RebaselineCycleConfig` (`crates/jurisearch-package-build/src/cycle.rs:258`, `crates/jurisearch-package-build/src/cycle.rs:260`, `crates/jurisearch-package-build/src/cycle.rs:261`). The embedded manifest does carry the rebaseline artifact's `baseline_id` (`crates/jurisearch-package-build/src/baseline.rs:377`, `crates/jurisearch-package-build/src/baseline.rs:387`), but the resume report drops it. In the producer, the current run derives `baseline_id` from the currently pending fetched baseline names (`crates/jurisearch-producer/src/update.rs:530`, `crates/jurisearch-producer/src/update.rs:546`) and, after `rebaseline_cycle` returns, adopts those current `new_baselines` unconditionally (`crates/jurisearch-producer/src/update.rs:277`, `crates/jurisearch-producer/src/update.rs:288`, `crates/jurisearch-producer/src/update.rs:290`). Repro shape: run A sees DILA baseline B1, ingests/builds a rebaseline for B1, and crashes after staging but before publish; before the retry, fetch advances the cursor to B2; run B routes for B2, ingests B2, then `rebaseline_cycle` resumes and publishes the old staged B1 artifact and stops; `run_update_inner` then records B2 as adopted. Status now reports no pending baseline, while the served manifest/package only re-anchored to B1 and may have been cut before B2 ingest. This loses the automatic rebaseline for B2. The resume path needs to either prove the resumed artifact corresponds to the baselines being adopted, or publish the old staged package without adopting the newer markers and then build/require a successor rebaseline for the still-pending baseline.

NIT: The new stale-by-age test has a false-green hole for the overall classification. `a_last_successful_but_old_run_classifies_as_stale_by_age` only creates a successful old record for `jurisprudence`; the example config still has `legislation` with no run, so `any_never_ran` would make the overall state stale even if `any_stale` were accidentally omitted from `OverallState::classify`'s `behind` input (`crates/jurisearch-producer/tests/run_record_and_status.rs:175`, `crates/jurisearch-producer/tests/run_record_and_status.rs:198`; production code uses `any_pending || any_running || any_never_ran || any_stale` at `crates/jurisearch-producer/src/status.rs:198`). The implementation is correct, but the test should seed non-stale successful records for every group and assert `status.overall == OverallState::Stale` specifically because of the old successful run.

## Notes

The original rebaseline pre-publish hole is fixed for the same-baseline retry case: the pending rebaseline is published before manifest rebuild, the same package id is used, and the manifest is generated only after the artifact exists. A staged incremental followed by rebaseline also looks source-correct: the incremental is resumed/marked first, then `build_rebaseline` chains from the now-published head.

The rebaseline routing read is now under the `update-core` lock, and the ordinary incremental guard is also checked under the lock before publishing. Relative path validation covers the systemd-rendered install paths and producer data/state paths. The status stale-by-age implementation correctly feeds `any_stale` into overall classification. Run ids no longer collide for the normal separate-process same-second timer/manual case because they include subsecond time and pid.

VERDICT: FIXES_REQUIRED
