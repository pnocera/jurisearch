# Codex Re-review: M1-B S2 migrations + S3 external DB provisioning

## Findings

### BLOCKER: Provisioned external writer still cannot run the advertised producer write path

The new `PRODUCER_PUBLIC_WRITE_TABLES` grant surface fixes the prior ingest-accounting omission for `index_manifest`, `schema_migrations`, `package_change_log`, `package_catalog`, `ingest_run`, `ingest_member`, and `ingest_error` ([crates/jurisearch-storage/src/generations.rs:109](/home/pierre/Work/jurisearch-worktrees/m1b-migrations-provision/crates/jurisearch-storage/src/generations.rs:109)). However, it is still not complete for the external producer contract this patch exposes. `DbClientSource` is documented as the seam that lets the producer run `ingest -> enrich -> embed -> package` against an external PostgreSQL through `ConnectionConfig`/`WriterHandle` ([crates/jurisearch-storage/src/backend.rs:181](/home/pierre/Work/jurisearch-worktrees/m1b-migrations-provision/crates/jurisearch-storage/src/backend.rs:181)), and those connections pin `search_path=public` specifically so unqualified producer helper SQL writes the producer working schema ([crates/jurisearch-storage/src/backend.rs:59](/home/pierre/Work/jurisearch-worktrees/m1b-migrations-provision/crates/jurisearch-storage/src/backend.rs:59)).

Those producer helper paths do write public replicated tables. For example, the LEGI projection prepares unqualified `INSERT INTO documents`, `INSERT INTO chunks`, and `INSERT INTO graph_edges` statements ([crates/jurisearch-storage/src/projection/legi.rs:49](/home/pierre/Work/jurisearch-worktrees/m1b-migrations-provision/crates/jurisearch-storage/src/projection/legi.rs:49), [crates/jurisearch-storage/src/projection/legi.rs:77](/home/pierre/Work/jurisearch-worktrees/m1b-migrations-provision/crates/jurisearch-storage/src/projection/legi.rs:77), [crates/jurisearch-storage/src/projection/legi.rs:101](/home/pierre/Work/jurisearch-worktrees/m1b-migrations-provision/crates/jurisearch-storage/src/projection/legi.rs:101)), and the official-API archive inserts into unqualified `official_api_responses` ([crates/jurisearch-storage/src/official_api_archive.rs:71](/home/pierre/Work/jurisearch-worktrees/m1b-migrations-provision/crates/jurisearch-storage/src/official_api_archive.rs:71)), which has a `bigserial` `response_id` sequence ([crates/jurisearch-storage/src/migrations.rs:614](/home/pierre/Work/jurisearch-worktrees/m1b-migrations-provision/crates/jurisearch-storage/src/migrations.rs:614)).

`provision_roles` explicitly revokes the writer's public table privileges, grants only `SELECT` on the replicated public templates, and then grants DML only on `PRODUCER_PUBLIC_WRITE_TABLES` ([crates/jurisearch-storage/src/backend.rs:552](/home/pierre/Work/jurisearch-worktrees/m1b-migrations-provision/crates/jurisearch-storage/src/backend.rs:552), [crates/jurisearch-storage/src/backend.rs:560](/home/pierre/Work/jurisearch-worktrees/m1b-migrations-provision/crates/jurisearch-storage/src/backend.rs:560), [crates/jurisearch-storage/src/backend.rs:570](/home/pierre/Work/jurisearch-worktrees/m1b-migrations-provision/crates/jurisearch-storage/src/backend.rs:570)). A freshly provisioned external writer can therefore pass the strengthened postcondition but still fail as soon as the real external producer tries to project a document, chunk, edge, API archive row, dense update, zone unit, citation, etc. It will also lack `USAGE` on replicated-table public sequences such as `official_api_responses_response_id_seq`.

Fix: decide whether S3 provisions an external producer database or only a client package-apply database. If it is the producer path described by `DbClientSource` and `provision_external_db`, the writer's public DML/sequence grant source must include every public table the producer mutates, including the replicated working tables and their owned sequences, and the postcondition/integration test should perform at least one representative replicated-table write that would currently fail. If the intended surface is only a client-side package applier, narrow the docs/types/reporting so they do not claim external producer ingest/enrich/embed support.

### WARN: Public sequence grants are additive, so a drifted writer can keep over-broad public sequence access

The table surface is converged by `REVOKE ALL ON ALL TABLES IN SCHEMA public FROM {writer}` before selected grants are rebuilt ([crates/jurisearch-storage/src/backend.rs:552](/home/pierre/Work/jurisearch-worktrees/m1b-migrations-provision/crates/jurisearch-storage/src/backend.rs:552)). The public sequence surface is not similarly converged: the code grants selected producer-owned sequences via the `pg_depend` DO block ([crates/jurisearch-storage/src/backend.rs:593](/home/pierre/Work/jurisearch-worktrees/m1b-migrations-provision/crates/jurisearch-storage/src/backend.rs:593)), but never revokes existing writer privileges on public sequences first. A deployment that previously had `GRANT USAGE, SELECT ON ALL SEQUENCES IN SCHEMA public TO jurisearch_write` will retain access to sequences the comments say must not be granted, such as replicated-table sequences.

Fix: before the selected public-sequence DO block, revoke the writer's public sequence privileges (`REVOKE ALL ON ALL SEQUENCES IN SCHEMA public FROM {writer}`), then grant back only the intended owned-sequence set. Also constrain the `pg_depend` query to the intended referenced table namespace, not just `t.relname`, so the selected set is exactly the public tables in the audited write surface.
