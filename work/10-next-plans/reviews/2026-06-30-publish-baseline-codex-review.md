## Findings

BLOCKER: A crash after the catalog row is marked `published` but before `core/manifest.json` is written leaves the one-shot operation unrecoverable by re-run. `bootstrap_build_locked` builds the first baseline, publishes the artifact, marks `core-1-1` as `published`, and clears staging before returning (`crates/jurisearch-package-build/src/cycle.rs:547`, `crates/jurisearch-package-build/src/cycle.rs:550`, `crates/jurisearch-package-build/src/cycle.rs:552`, `crates/jurisearch-package-build/src/cycle.rs:553`). Only afterward does `bootstrap_first_baseline` build and publish the remote manifest (`crates/jurisearch-package-build/src/cycle.rs:503`, `crates/jurisearch-package-build/src/cycle.rs:510`). If the process dies or `build_remote_manifest` / `publish_remote_manifest` fails in that interval, the DB contains a `published` first baseline and the artifact exists, but there is no served manifest. A retry cannot finalize it, because `bootstrap_preflight` refuses any existing baseline/rebaseline row regardless of whether the manifest is missing or invalid (`crates/jurisearch-package-build/src/cycle.rs:585`, `crates/jurisearch-package-build/src/cycle.rs:593`, `crates/jurisearch-package-build/src/cycle.rs:601`). This violates the required crash/retry behavior for the live one-time operation.

Concrete fix: make bootstrap finalization resumable. If the newest media row is the canonical first baseline (`core-1-1`, sequence 1, expected generation/baseline id/fingerprint) and the published artifact verifies against the row, allow the command to skip the build phase and rebuild/publish/verify `core/manifest.json`. Treat `status='built'` plus a matching staged or already-published artifact as either a resumable publish+mark+manifest path or a precise repair refusal. Add a fault seam/test for each boundary: after catalog insert, after `publish_package`, after `mark_package_published`, and after manifest temp write/rename.

BLOCKER: The code can make a bad remote manifest client-visible before the self-verification failure is reported. `build_remote_manifest` reads the published artifact and checks catalog identity (`crates/jurisearch-package-build/src/remote_manifest.rs:99`, `crates/jurisearch-package-build/src/remote_manifest.rs:101`), but `verify_catalog_identity` checks payload/catalog fields and the canonical embedded-manifest payload digest, not the embedded artifact signature (`crates/jurisearch-package-build/src/remote_manifest.rs:237`, `crates/jurisearch-package-build/src/remote_manifest.rs:244`, `crates/jurisearch-package-build/src/remote_manifest.rs:299`). `bootstrap_first_baseline` then renames the signed remote manifest into place before calling `verify_published_root` (`crates/jurisearch-package-build/src/cycle.rs:510`, `crates/jurisearch-package-build/src/cycle.rs:513`). A corrupted embedded signature, wrong verifier/trust-anchor wiring, or artifact mutation in this interval is only caught after `manifest.json` is already visible.

Concrete fix: verify the in-memory `Signed<RemoteManifest>` and every referenced artifact with the public verifier before the atomic rename, then optionally keep the post-rename `verify_published_root` as a final readback check. The clean shape is a shared helper that takes `&Signed<RemoteManifest>` instead of reading `manifest.json` from disk, verifies the remote signature, resolves each `artifact_uri`, verifies each embedded artifact signature, and checks the advertised digest. Add a regression that corrupts only the embedded artifact signature before remote-manifest publish and asserts no client-visible `core/manifest.json` is left behind.

WARN: The loopback test does not actually exercise the syncd manifest fetch/source path it claims to prove, so it can go green with a broken `artifact_uri` contract. The test reads `core/manifest.json` directly and verifies it manually (`crates/jurisearch-package-build/tests/bootstrap_baseline_loopback.rs:214`, `crates/jurisearch-package-build/tests/bootstrap_baseline_loopback.rs:216`), then applies through a custom `LocalSource` keyed by `baseline_id` / `package_id` (`crates/jurisearch-package-build/tests/bootstrap_baseline_loopback.rs:118`, `crates/jurisearch-package-build/tests/bootstrap_baseline_loopback.rs:124`, `crates/jurisearch-package-build/tests/bootstrap_baseline_loopback.rs:224`). That bypasses `DirectoryCatchupSource` URI resolution and `fetch_verify_manifest`, the client-facing path described in the review requirements. The producer wrapper test also only asserts the second run is some `ProducerError::Build` with class `publish-failed`, not the specific already-baselined refusal (`crates/jurisearch-producer/tests/publish_baseline_gated.rs:153`, `crates/jurisearch-producer/tests/publish_baseline_gated.rs:155`).

Concrete fix: in the package-build loopback, use `DirectoryCatchupSource::new(served.path(), URI_BASE)`, call `fetch_verify_manifest(&source, &client_verifier, "core")`, plan from that returned manifest, and pass the same source to `run_catchup`. Keep the digest convergence assertion. Tighten rejection tests to assert the specific diagnostic or structured failure for already-cataloged, partial `built`, schema mismatch, missing chunks, missing zones, and wrong fingerprint. Add the crash-recovery tests from the first finding so the current unrecoverable published-row/no-manifest state cannot regress silently.

## Confirmed

The lock deviation is sound as implemented. `run_publish_baseline` holds the same file-backed `update-core` lock across the external `WriterHandle` bootstrap (`crates/jurisearch-producer/src/publish_baseline.rs:91`, `crates/jurisearch-producer/src/publish_baseline.rs:93`, `crates/jurisearch-producer/src/publish_baseline.rs:109`), while normal `update` holds that lock across its DB-mutating ingest/embed/publish span (`crates/jurisearch-producer/src/update.rs:253`, `crates/jurisearch-producer/src/update.rs:255`, `crates/jurisearch-producer/src/update.rs:325`). Releasing the per-corpus advisory lock before `build_remote_manifest` is necessary because `build_remote_manifest` takes that same session advisory lock on its own connection (`crates/jurisearch-package-build/src/cycle.rs:494`, `crates/jurisearch-package-build/src/cycle.rs:497`, `crates/jurisearch-package-build/src/remote_manifest.rs:71`, `crates/jurisearch-package-build/src/remote_manifest.rs:72`). The `update-core` lock remains the cross-run serializer, so a timer update cannot race the bootstrap.

The canonical first-baseline identity is correct: `build_baseline` emits `core-1-1`, sequence 1, generation `core_g0001`, `package_kind=baseline`, and `included_change_seq_low=0` (`crates/jurisearch-package-build/src/baseline.rs:151`, `crates/jurisearch-package-build/src/baseline.rs:157`, `crates/jurisearch-package-build/src/baseline.rs:158`, `crates/jurisearch-package-build/src/baseline.rs:162`). The producer wrapper rejects the `jurisearch-package` CLI `:cls:` fingerprint and uses the config storage fingerprint (`crates/jurisearch-producer/src/publish_baseline.rs:75`, `crates/jurisearch-producer/src/publish_baseline.rs:76`, `crates/jurisearch-producer/src/publish_baseline.rs:148`).

The preflights are pointed at real producer rows and cover the important embedding holes: chunk and zone embedding joins require fingerprint/model/dimension matches, and the source tables must have the expected fingerprint stamped (`crates/jurisearch-package-build/src/cycle.rs:612`, `crates/jurisearch-package-build/src/cycle.rs:633`, `crates/jurisearch-package-build/src/cycle.rs:648`, `crates/jurisearch-package-build/src/cycle.rs:668`). The schema check is tied to `CURRENT_SCHEMA_VERSION` (`crates/jurisearch-package-build/src/cycle.rs:568`, `crates/jurisearch-package-build/src/cycle.rs:574`).

The status-filter deviation is acceptable for the narrow bootstrap path because the row is marked `published` before the manifest is built (`crates/jurisearch-package-build/src/cycle.rs:550`, `crates/jurisearch-package-build/src/cycle.rs:552`, `crates/jurisearch-package-build/src/cycle.rs:503`). However, `build_remote_manifest` still selects catalog rows without filtering on `status` (`crates/jurisearch-package-build/src/remote_manifest.rs:93`, `crates/jurisearch-package-build/src/remote_manifest.rs:121`, `crates/jurisearch-package-build/src/remote_manifest.rs:123`), so the recovery fixes above should be landed before this is trusted for the live production bootstrap.
