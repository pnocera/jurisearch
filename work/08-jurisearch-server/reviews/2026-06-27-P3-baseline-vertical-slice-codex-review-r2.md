## Findings

### WARN 1 - The client cursor still records the manifest digest as the package digest

The r1 catalog fix correctly separates the payload/package digest from the manifest digest on the producer side: `build_baseline` computes `payload_digest`, puts it in `manifest.integrity.artifact_sha256`, and writes it to `package_catalog.package_digest`, while separately writing `manifest_digest` (`crates/jurisearch-package-build/src/baseline.rs:245`, `crates/jurisearch-package-build/src/baseline.rs:285`, `crates/jurisearch-package-build/src/baseline.rs:304`). But the consumer side still treats the canonical manifest digest as the package digest. `apply_baseline` computes `manifest_digest`, uses it for the idempotency decision, and stamps it into `corpus_state.last_package_digest`/`generation_registry.validation_digest` (`crates/jurisearch-syncd/src/apply.rs:87`, `crates/jurisearch-syncd/src/apply.rs:89`, `crates/jurisearch-syncd/src/apply.rs:126`). `idempotency_decision` then reads only `sequence, last_package_digest` and compares that value to the manifest digest (`crates/jurisearch-syncd/src/apply.rs:340`, `crates/jurisearch-syncd/src/apply.rs:352`).

That leaves two digest meanings in the system immediately after the baseline: the producer catalog's `package_digest` is the payload/artifact digest, while the client cursor's `last_package_digest` is the embedded-manifest digest. The P3 no-op reapply still works only because both write and read the same mistaken cursor value. The next phase's chain link (`previous_package_sha256` / `last_package_digest`) will either compare producer package digests against a client manifest digest and reject a valid incremental, or force later code to keep carrying the same name with incompatible semantics. Also, once the cursor digest is corrected to the payload/package digest, the current idempotency check must include `last_package_id`; otherwise two different package IDs with identical payload bytes would look idempotent at the same sequence.

Actionable fix: keep `manifest_digest` as a separate verification/internal value, but derive a `package_digest` for cursor identity from `manifest.integrity.artifact_sha256` for P3 (and the real artifact digest once transport exists). Stamp `corpus_state.last_package_digest` and `generation_registry.validation_digest` with that package digest, and make `idempotency_decision` compare both `last_package_id == manifest.identity.package_id` and `last_package_digest == package_digest`.

### WARN 2 - The signed IVFFlat index contract is still only partially enforced

The producer declares IVFFlat finalization parameters in the signed manifest (`crates/jurisearch-package-build/src/baseline.rs:189`, `crates/jurisearch-package-build/src/baseline.rs:270`). The consumer rebuilds IVFFlat indexes with locally recomputed `lists` and records those values in `GenerationIndexReport` (`crates/jurisearch-storage/src/generations.rs:355`, `crates/jurisearch-storage/src/generations.rs:363`, `crates/jurisearch-storage/src/generations.rs:370`). But `validate_index_contract` ignores the declared `lists`/target shape and only checks that an index with the declared name exists in the generation with access method `ivfflat` or `bm25` (`crates/jurisearch-syncd/src/apply.rs:239`, `crates/jurisearch-syncd/src/apply.rs:246`, `crates/jurisearch-syncd/src/apply.rs:249`).

That resolves the "missing declared index" part of r1, but not the signed index-build contract. A consumer bug or drift that creates `chunk_embeddings_embedding_ivfflat_idx` with the wrong `lists`, or on the wrong relation/column but with the expected name and access method, would still pass validation and activate the generation. Postcondition digests do not catch this because they validate table contents, not index shape.

Actionable fix: validate each `manifest.apply.index_build.ivfflat_finalize` entry against the actual generation index before `activate_generation`: verify the index relation, table, indexed column/operator class, access method, and `reloptions` `lists` value (or compare directly against a strengthened `GenerationIndexReport`). For BM25, at least verify the declared index is attached to one of the expected generation tables, or extend the manifest contract to carry the table target and check it here.
