### NIT: Keep "distinct server schema" wording clearly future-tense

The bottom line groups "a distinct server schema" with source-backed primitives such as deterministic row ids, timestamps, fingerprints, builder/schema versions, and the `SchemaVersionAhead` guard (`work/08-jurisearch-server/2026-06-25-central-ingest-delta-sync-analysis.md:413-425`). ## Claims Verified Against Source

- Core storage tables and columns: confirmed. `documents` has `document_id`, temporal fields, `source_payload_hash`, `created_at`, and `updated_at`; `chunks` has `source_payload_hash`, `chunk_builder_version`, `embedding_fingerprint`, and `created_at`; `chunk_embeddings` has `embedding_fingerprint`, `embedding vector(1024)`, `model`, `dimension = 1024`, and `created_at` (`crates/jurisearch-storage/src/migrations.rs:27-67`).
- `pg_search` / BM25 and pgvector extension setup: confirmed. Migrations create both extensions (`crates/jurisearch-storage/src/migrations.rs:23-25`), BM25 indexes use `USING bm25` (`crates/jurisearch-storage/src/migrations.rs:103-105`, `crates/jurisearch-storage/src/migrations.rs:355-369`, `crates/jurisearch-storage/src/migrations.rs:559-573`), and the runtime requires `pg_search` and `vector` assets before starting Postgres (`crates/jurisearch-storage/src/runtime.rs:99-101`, `crates/jurisearch-storage/src/runtime.rs:175-176`).
- IVFFlat finalize with caller-specified lists: confirmed. Chunk dense finalize drops/recreates `chunk_embeddings_embedding_ivfflat_idx` with `WITH (lists = spec.index_lists)` (`crates/jurisearch-storage/src/dense.rs:93-160`); zone-unit dense finalize does the same for `zone_unit_embeddings_ivfflat_idx` (`crates/jurisearch-storage/src/zone_units.rs:431-510`).

- `official_api_responses` archive shape: confirmed. Migration v16 creates `official_api_responses` with provider/endpoint/request fields, raw `response_body`, parsed `response_json`, and `response_body_sha256`, with comments explicitly describing byte-faithful body storage plus parsed jsonb and sha256 (`crates/jurisearch-storage/src/migrations.rs:582-637`).
- Derived-table delete/rebuild paths: confirmed. `replace_zone_units_for_document` deletes all units for a decision before reinserting the new derivation (`crates/jurisearch-storage/src/zone_units.rs:120-169`); `upsert_decision_zones_with_client` deletes `zone_units` when a refreshed zone row is not derivable (`crates/jurisearch-storage/src/decision_zones.rs:195-207`); hierarchy backfill deletes affected `chunk_embeddings` and clears chunk fingerprints (`crates/jurisearch-storage/src/projection/hierarchy_backfill.rs:209-229`).
- No uniform `updated_at` watermark across replicated tables: confirmed. `documents` has `updated_at`, but `chunks`, `chunk_embeddings`, `zone_units`, and `zone_unit_embeddings` shown in migrations have `created_at` only; LEGI chunk upserts update fields without stamping `updated_at` (`crates/jurisearch-storage/src/migrations.rs:27-67`, `crates/jurisearch-storage/src/migrations.rs:495-539`, `crates/jurisearch-storage/src/projection/legi.rs:71-92`).
- Temporal/as-of model: confirmed. - `document_id` stability claim: confirmed only in the document's narrowed sense. The validator requires LEGI `document_id == legi:<source_uid>@<valid_from>` (`crates/jurisearch-ingest/src/legi/canonical.rs:52-56`), tests show `version_group` is the underlying `LEGIARTI...` family id (`crates/jurisearch-ingest/src/legi/tests.rs:18-27`), and projection upserts on `document_id` while updating temporal and payload fields on conflict (`crates/jurisearch-storage/src/projection/legi.rs:45-68`). - Existing `jurisearch sync`: confirmed as local official-source archive delta ingestion, not server-to-client package distribution. `sync_payload` builds an `ArchiveSyncFilter { incremental: true, since_compact }` and calls the normal LEGI/JURI archive ingest paths (`crates/jurisearch-cli/src/ingest.rs:24-89`), while the CLI still labels `sync` as a STUB (`crates/jurisearch-cli/src/args.rs:135-138`).
