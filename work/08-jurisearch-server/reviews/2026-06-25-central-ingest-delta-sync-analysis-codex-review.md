The analysis is directionally sound for the central-producer/read-only-client architecture: the schema really is ordinary PostgreSQL with `vector` and `pg_search`, the client `serve` process is only a local query daemon today, the read-only split correctly identifies query-time enrichment writes, and the three replication mechanisms are mostly characterized with the right tradeoffs. It should not become the analysis of record unchanged, though, because it overstates the corpus as append-only/low-delete: base legal documents are temporal and upserted, but derived retrieval tables have routine delete/rebuild paths that any replication design must handle, and the document should also clarify that `crates/` already contains local official-source archive `sync` code even though it has no server-to-client replication system.

## Findings

### WARN - Derived retrieval rows are routinely deleted/rebuilt, so the "only rare row removals" replication claim is too broad

The analysis says the change stream has "no need to reconcile vanished rows" and that "the only genuine row removals are rare administrative cases" (`work/08-jurisearch-server/2026-06-25-central-ingest-delta-sync-analysis.md:67-73`). That is true enough for superseded LEGI article versions, but not for the full replicated set the document lists. `zone_units` are replaced by deleting all units for a decision and reinserting the current derivation (`crates/jurisearch-storage/src/zone_units.rs:120-145`), `decision_zones` refresh invalidation deletes already-materialized `zone_units` for non-derivable rows (`crates/jurisearch-storage/src/decision_zones.rs:195-204`), and hierarchy backfill can delete `chunk_embeddings` while clearing chunk fingerprints (`crates/jurisearch-storage/src/projection/hierarchy_backfill.rs:209-229`). Those are normal maintenance/rebuild paths, not rare admin redactions.

Recommended fix: narrow the append-only claim to base legal documents/articles and explicitly state that derived/index-support tables (`zone_units`, `zone_unit_embeddings`, `chunk_embeddings`, possibly `decision_zones`) need delete/replace events or generation-scoped rebuild semantics in logical replication and application-level bundles.

### WARN - The document should distinguish "no server-to-client replication" from the existing local archive `sync`

The analysis treats the operated replication/distribution layer as net-new, which is fair for server-to-client replication, but the brief asks to verify the claim that no sync/delta/replication code exists today. That broad wording is false: `sync_payload` implements incremental official-source archive sync from local delta archives (`crates/jurisearch-cli/src/ingest.rs:24-89`), the CLI advertises `sync` as "Synchronize official sources through deltas or transactional histories (STUB)" (`crates/jurisearch-cli/src/args.rs:135-138`), and `ArchiveSyncFilter` selects delta archives without re-reading the baseline (`crates/jurisearch-cli/src/ingest.rs:330-358`). This is not the proposed central server replication mechanism, but leaving it unnamed makes the analysis easier to misread.

Recommended fix: add one sentence under "Remaining risks" or "Replication mechanism" saying existing `jurisearch sync` is local official-source archive delta ingestion only; server-to-client corpus replication, entitlement, schema distribution, bundle/WAL endpoints, and version gating remain unimplemented.

### NIT - The IVFFlat/BM25 replication framing should separate repo evidence from external PostgreSQL/extension evidence

The repo confirms the local facts: migrations create `vector` and `pg_search` (`crates/jurisearch-storage/src/migrations.rs:24-25`), BM25 indexes use `USING bm25` (`crates/jurisearch-storage/src/migrations.rs:355-369`, `crates/jurisearch-storage/src/migrations.rs:559-573`), and IVFFlat indexes are built at finalize with caller-specified `lists` (`crates/jurisearch-storage/src/dense.rs:151-160`, `crates/jurisearch-storage/src/zone_units.rs:489-498`). The repo does not prove the external claim that pgvector IVFFlat is safe under physical replication, nor does it prove/refute ParadeDB BM25 block-replication behavior; it only shows `pg_search` is a custom extension that must be preloaded (`crates/jurisearch-storage/src/runtime.rs:454-456`) and queried through `paradedb.score`/`@@@` (`crates/jurisearch-storage/src/retrieval/sql.rs:79-90`).

Recommended fix: keep the current open question, but phrase it as "repo confirms both index types are extension indexes; physical-replication safety for pgvector IVFFlat and especially pg_search BM25 must be verified against upstream extension/PostgreSQL docs or a physical-standby smoke test."
