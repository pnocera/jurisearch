## Findings

### BLOCKER: `jurisearch-package verify` can pass even when the published remote manifest is missing or stale

The new `Verify` command does not read or verify the published `<root>/<corpus>/manifest.json` at all. It re-runs `build_remote_manifest` from the catalog and artifact directories, signs that in-memory manifest with `--signing-seed-hex`, and prints OK (`crates/jurisearch-package-build/src/bin/jurisearch_package.rs:268-284`). That means an operator can delete, corrupt, or leave stale the actual manifest clients poll, run `jurisearch-package verify`, and still get `OK` as long as the cataloged artifact directories are present. The command also requires the private signing seed for a read-only verification action (`crates/jurisearch-package-build/src/bin/jurisearch_package.rs:89-99`, `crates/jurisearch-package-build/src/bin/jurisearch_package.rs:271-278`), which is the wrong key-handling shape even with KMS/rotation deferred.

Concrete fix: make `verify` a read-only published-root verifier. It should read `published_manifest_path(root, corpus)`, verify the signed `RemoteManifest` with a public trust anchor/verifier, check the manifest corpus matches the requested corpus, and then verify every referenced artifact exists and matches its remote `sha256`/signature/catalog identity. If you also want a "can I build the next manifest?" check, keep that as a separate `verify-catalog` or compare the existing manifest payload to a newly built expected payload. Add a regression that deletes or tampers `root/core/manifest.json` after `publish-manifest` and proves `jurisearch-package verify` fails.

### BLOCKER: The remote-manifest builder does not validate the embedded manifest against the cataloged manifest identity

`package_catalog` records both `package_digest` and `manifest_digest` when artifacts are built (`crates/jurisearch-package-build/src/baseline.rs:465-490`, `crates/jurisearch-package-build/src/incremental.rs:501-523`), but the P9 read model drops `manifest_digest` from `CatalogRow` (`crates/jurisearch-storage/src/package_catalog.rs:181-196`) and the query (`crates/jurisearch-storage/src/package_catalog.rs:207-212`). The remote-manifest builder only checks `row.package_digest == embedded.integrity.artifact_sha256` (`crates/jurisearch-package-build/src/remote_manifest.rs:225-234`). Because `artifact_sha256` is the payload aggregate, a published `manifest.json` can have a changed package id, kind, sequence, compatibility fields, postconditions, or signature while keeping the same payload digest, and `build_remote_manifest` will still sign remote metadata derived from that changed embedded manifest (`crates/jurisearch-package-build/src/remote_manifest.rs:97-151`).

That leaves a real false-green hole in the "manifest build fails on mismatched artifact before any client sees it" gate. A malformed or wrongly copied embedded manifest with the same payload bytes is a mismatched artifact, but the producer-side verify path accepts it and may publish a remote manifest that clients only reject later, or accept if it was accidentally re-signed with the producer key.

Concrete fix: include `manifest_digest`, `baseline_id`, `generation`, `package_kind`, and `package_sequence` in the catalog read checks. Recompute `canonical_digest(&signed.payload)` for each embedded manifest and require it to match `row.manifest_digest`; also require `identity.package_id`, `identity.package_kind`, `identity.baseline_id`, `identity.generation`, and `identity.to_sequence` to match the catalog row. For retained incrementals, additionally require `identity.package_kind == Incremental` and `identity.to_sequence == identity.from_sequence.next()`. Add a regression that edits only the embedded manifest identity/compatibility while leaving payload files unchanged and proves `build_remote_manifest` fails.

### WARN: `syncd update --corpus` can apply a signed manifest for a different corpus

`Update` chooses the file path from the requested `--corpus`, verifies the signed remote manifest, then immediately plans from `signed.payload` without checking `signed.payload.corpus` against the requested corpus (`crates/jurisearch-syncd/src/main.rs:128-141`). The status messages still use the CLI `corpus` string (`crates/jurisearch-syncd/src/main.rs:142-156`), but the apply path uses the embedded manifest's corpus when loading media packages (`crates/jurisearch-syncd/src/apply.rs:171-180`) and incrementals (`crates/jurisearch-syncd/src/apply.rs:808-815`). If `root/core/manifest.json` is stale or misplaced but signed for `inpi`, `jurisearch-syncd --index-dir ... update --corpus core` can report a core update while installing or advancing `inpi`.

Concrete fix: after remote-manifest signature verification, reject unless `signed.payload.corpus.as_str() == corpus`. Ideally also assert the active baseline and package artifact URIs resolve under the same corpus path shape for the configured `uri_base`. Add a CLI/update test with a signed wrong-corpus manifest under the requested path and assert no corpus is installed or advanced.

### WARN: `publish_package` is not atomic when replacing an existing package directory

The publisher stages to a sibling `.tmp` directory, but when the destination already exists it removes the live package directory before renaming staging into place (`crates/jurisearch-package-build/src/publish.rs:45-53`). That creates a window where a manifest that already references that immutable package id points at no directory. It also means the documented "replacing any prior copy" behavior is not an atomic directory replacement on the filesystem.

Concrete fix: treat published package ids as immutable. If `dest` already exists, verify it is byte-for-byte/catalog-identical and return it as an idempotent no-op, or fail with a clear "package already published" error. For a genuinely changed rebuild, publish under a new package id and then atomically update `manifest.json`. Keep `manifest.json` as the only in-place replacement surface.
