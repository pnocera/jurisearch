The analysis is substantially source-grounded and internally much stronger than the architecture sketch it reviews: it correctly separates base temporal rows from derived rebuild tables, correctly narrows "`document_id` never changes" to immutable version-specific row IDs, and carries the main consequences of ordered per-corpus package chains. I do not think it is ready to stand as the analysis of record yet, because it still leaves one important PostgreSQL packaging option overstated and describes the current schema's change-tracking primitives more uniformly than the repo supports.

## Findings

### WARN - Prebuilt PostgreSQL index packages are treated as an ordinary retained option, but that reintroduces the native-replication constraints the analysis rejects.

The document rejects physical/block-level standby partly because it is whole-cluster, tied to major version/CPU architecture, and has unverified custom-index safety for `pgvector` IVFFlat and `pg_search` BM25 (`work/08-jurisearch-server/2026-06-25-central-ingest-delta-sync-analysis.md:175-181`). Later it says a package can either carry rows+DDL and build indexes on the client, or "ship prebuilt index data" with no client build, retaining both options (`work/08-jurisearch-server/2026-06-25-central-ingest-delta-sync-analysis.md:194-199`). The source shows these are ordinary PostgreSQL extension indexes: BM25 is created with `USING bm25` in migrations (`crates/jurisearch-storage/src/migrations.rs:355-369`, `crates/jurisearch-storage/src/migrations.rs:559-573`), and IVFFlat is created by `CREATE INDEX ... USING ivfflat` during finalize with caller-specified `lists` (`crates/jurisearch-storage/src/dense.rs:151-160`, `crates/jurisearch-storage/src/zone_units.rs:489-498`). PostgreSQL does not provide a portable logical artifact for a populated custom index independent of the table data; copying relation files would bring back the same engine/version/extension-binary compatibility and trust issues that motivated rejecting the physical option. Recommended fix: make "client builds indexes from packaged rows+DDL" the default analysed path, and describe "prebuilt indexes" as a high-risk physical-format variant that needs explicit same-version/extension/architecture constraints, not as a peer option retained by the generic package format.

### WARN - The current schema does not provide uniform timestamp-based change tracking across the replicated tables.

The analysis lists "source_payload_hash on documents and chunks, `created_at`/`updated_at`, `embedding_fingerprint` / `chunk_builder_version` / `zone_unit_builder_version` / `model`" as schema primitives useful to any mechanism (`work/08-jurisearch-server/2026-06-25-central-ingest-delta-sync-analysis.md:130-136`). The individual columns are real in places, but not uniform: `documents` has both `created_at` and `updated_at` (`crates/jurisearch-storage/src/migrations.rs:27-44`), `chunks` has `created_at` but no `updated_at` (`crates/jurisearch-storage/src/migrations.rs:46-58`), `chunk_embeddings` has `created_at` only (`crates/jurisearch-storage/src/migrations.rs:60-67`), `zone_units` has `created_at` only (`crates/jurisearch-storage/src/migrations.rs:495-511`), and `zone_unit_embeddings` has `created_at` only (`crates/jurisearch-storage/src/migrations.rs:532-539`). The LEGI chunk upsert updates body/provenance/fingerprint fields but has no chunk `updated_at` to stamp (`crates/jurisearch-storage/src/projection/legi.rs:73-92`). This matters because the package producer cannot safely be described as having generic `updated_at` watermarks for all replicated data. Recommended fix: say the schema has useful identity/hash/fingerprint/builder-version fields, but package generation still needs an explicit per-table diff ledger or snapshot comparison, especially for chunk and derived-table updates.

### NIT - "A schema migration" is overstated as always requiring a full re-baseline.

The document says a corpus-invalidating change - "a re-embed, a builder-version bump, or a schema migration" - cannot be expressed as a diff and must be handled by a new physical-media baseline (`work/08-jurisearch-server/2026-06-25-central-ingest-delta-sync-analysis.md:210-217`). That is right for breaking or corpus-rewriting migrations, but too broad for additive DDL because the same document says packages carry schema with data (`work/08-jurisearch-server/2026-06-25-central-ingest-delta-sync-analysis.md:40-41`, `work/08-jurisearch-server/2026-06-25-central-ingest-delta-sync-analysis.md:162-163`). Recommended fix: narrow the sentence to "breaking or corpus-invalidating schema migrations" and leave additive schema changes as package-carried migrations when compatible with the client version gate.
