Overall judgement: the analysis is directionally strong on the selected package model and gets most repo-grounded facts right, especially schema version 17, pgvector/pg_search usage, lazy-vs-upfront enrichment consequences, derived-table rebuild/delete paths, and the fact that the current `jurisearch sync` is unrelated local official-source delta ingestion. It is not sound enough to stand unchanged, because it overextends the new `document_id never changes` decision into a factual claim about logical article identity and writable-layer cross-reference safety that the current LEGI model refutes: article `document_id`s are version-specific, while `source_uid`/`version_group` are the stable logical article identity. It also understates the database mechanics required to preserve writable client tables that hold foreign keys into a server-managed schema during scoped reload/re-baseline.

## Findings

### BLOCKER: The `document_id never changes` framing is wrong for logical article versions and weakens the writable-layer safety argument.

The document repeatedly uses `document_id` as if it were a stable logical article identifier across corpus evolution: it says cross-references are safe because "`document_id` never changes" and the base corpus is append-only, so writable rows referencing a server row by `document_id` keep resolving across updates/re-baselines (`work/08-jurisearch-server/2026-06-25-central-ingest-delta-sync-analysis.md:229-241`, `:303-306`, `:325-330`, `:369-371`). That is not how LEGI article identity is modeled today. The canonical article validator requires `document_id == format!("legi:{}@{}", source_uid, valid_from)` (`crates/jurisearch-ingest/src/legi/canonical.rs:52-56`), the parser constructs that exact ID (`crates/jurisearch-ingest/src/legi/parser.rs:557-562`), and the parser test confirms an article ID like `legi:LEGIARTI000006419320@1804-02-21` while `version_group` remains `LEGIARTI000006419320` (`crates/jurisearch-ingest/src/legi/tests.rs:18-27`). A new article version is therefore a new row with a new version-specific `document_id`; the logical article identity is `source_uid`/`version_group`, not `document_id`.

Recommended fix: rewrite the analysis to distinguish "row identity is immutable once created" from "logical article identity is stable." Cross-reference guidance should say writable rows that need an exact historical version may reference `documents.document_id`, but writable rows that mean "this logical article across versions" must reference `source_uid`/`version_group` plus an as-of date or their own version-selection policy. The bottom-line and decided/deferred ledger should remove or qualify bare "`document_id` never changes" so it does not imply one logical article has one forever ID.

### WARN: Scoped re-baseline plus writable foreign keys needs explicit constraint mechanics.

The document says a writable client table may FK to a read-only server row by `document_id` and that a re-baseline can drop/reload the server set while preserving writable tables (`work/08-jurisearch-server/2026-06-25-central-ingest-delta-sync-analysis.md:229-238`, `:299-306`). That is directionally plausible only with a careful procedure. In PostgreSQL, a writable table with a foreign key into a server-managed schema creates a dependency on the referenced server table; dropping/reloading that schema can be blocked, can require dropping/recreating constraints, or can accidentally cascade effects depending on the chosen DDL/DML path. The current server tables themselves use cascading references internally, e.g. `chunks.document_id REFERENCES documents(document_id) ON DELETE CASCADE`, `graph_edges` cascades, `decision_zones` cascades, and `zone_units` cascades (`crates/jurisearch-storage/src/migrations.rs:46-76`, `:450-451`, `:495-511`), which is fine inside the server set but highlights why a client-owned FK boundary must be designed deliberately.

Recommended fix: add a risk/design requirement that scoped reload must define the cross-schema FK strategy. Viable options include disallowing direct writable FKs into server tables and using validated soft references, loading the new server schema under a new generation and atomically switching a view/search path, or temporarily dropping and recreating client-owned constraints after validating references. The analysis should not state that FK cross-references are "safe by construction" until that mechanism is named.

### WARN: The document calls base-corpus replication "monotonic and mostly additive" but should explicitly include updates to existing version rows, not just inserts.

The text correctly says a new article version is a new row plus a validity-window update on the old row (`work/08-jurisearch-server/2026-06-25-central-ingest-delta-sync-analysis.md:82-87`), but the next paragraph leans hard on monotonic/additive replication and "idempotent upsert by stable source-namespaced PK" (`:89-94`). Current projection upserts by `document_id` and updates `valid_to`, `source_payload_hash`, `canonical_json`, `updated_at`, and other fields on conflict (`crates/jurisearch-storage/src/projection/legi.rs:45-68`). The package format therefore needs row-level updates for existing version-specific document rows, not only append events for new rows and delete/replace events for derived tables.

Recommended fix: in the incremental-package requirements, explicitly list base-table update events for closed validity windows and source corrections. Keep the "no deletion feed for supersession" point, but avoid implying append-only alone is sufficient for base `documents`.

### NIT: "Stable source-namespaced PKs" is too broad unless it distinguishes decisions from LEGI articles.

The analysis refers to stable source-namespaced PKs (`work/08-jurisearch-server/2026-06-25-central-ingest-delta-sync-analysis.md:89-91`, `:126-129`, `:181-185`). For decisions, the validator requires `<source>:<source_uid>` (`crates/jurisearch-ingest/src/juri/types.rs:180-184`), so that shorthand works. For LEGI articles, the primary key is `<source>:<source_uid>@<valid_from>` (`crates/jurisearch-ingest/src/legi/canonical.rs:52-56`). The source-namespaced prefix is stable, but the full article row PK is version-specific.

Recommended fix: replace the generic phrase with "stable row PKs; decisions use `<source>:<source_uid>`, LEGI article versions use `legi:<source_uid>@<valid_from>`, and logical article families use `source_uid`/`version_group`."
