# Review: reassign-corpus-ownership.sql (r2)

## Findings

- BLOCKER: [work/12-db-ownership-fix/reassign-corpus-ownership.sql](/home/pierre/Work/jurisearch/work/12-db-ownership-fix/reassign-corpus-ownership.sql:64) orders the ownership loop by raw `relkind`, so `S` sequences are altered before `r` tables. That fails on any `bigserial`/owned sequence because PostgreSQL rejects `ALTER SEQUENCE ... OWNER` while the sequence is still linked to a differently owned table. The current schema has public `bigserial` columns, for example `ingest_member.member_id` and `ingest_error.error_id` in [crates/jurisearch-storage/src/migrations.rs](/home/pierre/Work/jurisearch/crates/jurisearch-storage/src/migrations.rs:156), `official_api_responses.response_id` in [crates/jurisearch-storage/src/migrations.rs](/home/pierre/Work/jurisearch/crates/jurisearch-storage/src/migrations.rs:615), and `package_change_log.change_seq` in [crates/jurisearch-storage/src/migrations.rs](/home/pierre/Work/jurisearch/crates/jurisearch-storage/src/migrations.rs:854). I reproduced this on PostgreSQL 18.4 with a `postgres`-owned `official_api_responses(response_id bigserial)` table: the script aborts inside the DO block with `cannot change owner of sequence "official_api_responses_response_id_seq"` before any table ownership is reassigned. Fix: split the loop so tables/partitioned tables are altered first, then re-query and alter only remaining `postgres`-owned standalone sequences; or explicitly exclude auto-dependent sequences from the sequence pass with `pg_depend` (`deptype = 'a'`), since `ALTER TABLE ... OWNER` moves linked indexes and owned sequences with the table.
