## Findings

BLOCKER: The new DDL probe can pass while the real dense-finalize DDL still fails on table ownership. `probe_role_can_create_in_public` creates its throwaway index on `public.index_manifest (key)` (`crates/jurisearch-storage/src/provision.rs:349`), and `build_provision_sql` explicitly transfers `public.index_manifest` to `jurisearch_owner` (`crates/jurisearch-storage/src/backend.rs:529`). The real producer DDL creates indexes on `public.chunk_embeddings` and `public.zone_unit_embeddings` (`crates/jurisearch-storage/src/dense.rs:253`, `crates/jurisearch-storage/src/zone_units.rs:687`), but the provisioning SQL does not transfer those public working tables to `jurisearch_owner`; the only public ownership transfers are `index_manifest`, `schema_migrations`, `package_change_log`, and `package_catalog` (`crates/jurisearch-storage/src/backend.rs:529`). PostgreSQL `CREATE INDEX` requires the caller to be the table owner or a member of the table-owning role, in addition to being able to create the index object in the schema. So this postcondition checks the new schema-level `CREATE` grant against a table that is easier than the actual target and can false-positive for the dense/zone rebuild path. Fix: either reassign the producer DDL target public working tables/indexes to `jurisearch_owner` during producer provisioning, or change the probe to create rollback-only redundant indexes on the actual public dense target table(s) after confirming the intended ownership model. Add a PG-gated regression that connects as `jurisearch_write` after `provision_external_db` and runs a rolled-back `CREATE INDEX` on `public.chunk_embeddings` at minimum, and preferably `public.zone_unit_embeddings` too.
