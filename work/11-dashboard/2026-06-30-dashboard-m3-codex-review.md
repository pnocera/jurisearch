## Findings

- BLOCKER: The bind guard is bypassable by wildcard spellings that `Bun.serve` accepts. `assertExplicitBind` is a literal deny-list in `apps/dashboard/server/src/http/bind.ts:10`, and the tests only cover the common forms in `apps/dashboard/server/src/http/router.test.ts:344`. I verified Bun accepts additional all-interface spellings such as `[0:0:0:0:0:0:0:0]`, `::ffff:0.0.0.0`, and `000.000.000.000`; none are rejected before `Bun.serve` at `apps/dashboard/server/src/http/router.ts:153`. A production misconfiguration using one of those forms can still expose the unauthenticated dashboard beyond the tailnet. Concrete fix: normalize and classify IP literals before the deny-list check (strip IPv6 brackets, lower-case, parse IPv4 including zero-padded all-zero forms, parse IPv6 including IPv4-mapped unspecified addresses) and reject every unspecified/all-zero address. Add `assertExplicitBind` and `startServer` tests for those spellings so the fail-closed contract proves no bind occurs.

- WARN: The composition root does not wire provider-level caches before services, so several configured TTLs have no effect. `main.ts` constructs `OverviewService` with raw `statusProvider`, `runsProvider`, and `timersProvider` at `apps/dashboard/server/main.ts:71`, then only caches the whole overview plus packages/runs/logs at `apps/dashboard/server/main.ts:75`. That leaves `config.cache.statusMs` and `config.cache.timersMs` unused, and `/api/overview` bypasses the `/api/runs` cache even though the service seam documents raw providers -> `Cached` -> services in `apps/dashboard/server/src/services/types.ts:4`. This creates duplicate provider I/O under normal dashboard polling and makes user-supplied source TTLs misleading. Concrete fix: wrap `StatusProvider`, `RunsProvider`, `PackagesProvider`, `LogsProvider`, and `TimersProvider` in `Cached` at the composition root, inject cached `DataProvider` adapters into `OverviewService`, and expose the same cached runs/logs/packages providers through `DashboardServices`. Keep an optional overview-level cache only if the design still wants page-level dedupe. Add a composition test with counting providers to prove each TTL knob is consumed.

- WARN: The overview last-run fallback can merge fields from two different runs. In `apps/dashboard/server/src/services/OverviewService.ts:105`, if `group.lastRunId` is non-null but the record is absent, the service falls back to the most recent record for the group; then `buildLastRun` reports the status `runId`/`outcome`/`exitClass`/`endedAt` together with the fallback record's `kind`/`startedAt` at `apps/dashboard/server/src/services/OverviewService.ts:137`. That can mislabel the run and compute a bogus duration when the status summary and fallback record are not the same run. Concrete fix: when `lastRunId` is set, only join a record with that exact id; if it is missing, keep the status fields and set record-only fields (`kind`, `startedAt`, `durationMs`) to null. Use the most-recent fallback only when status has no run id, or else derive the entire fallback last-run from the fallback record consistently. Add a regression test where status references a missing `lastRunId` while records contain another run.

- NIT: Numeric parsing accepts partial numbers. `parsePort` in `apps/dashboard/server/src/config/resolve.ts:124`, `parsePort` in `apps/dashboard/server/src/config/flags.ts:30`, and `positiveInt` in `apps/dashboard/server/src/http/router.ts:60` all use `Number.parseInt`, so values like `--port=123abc`, `DASHBOARD_PORT=123abc`, or `?limit=10junk` are accepted as valid numbers. Concrete fix: use an anchored decimal-integer check such as `/^\d+$/` before `Number(value)`, then apply the existing range checks; add parser tests for suffixed and fractional inputs.

VERDICT: FIXES_REQUIRED
