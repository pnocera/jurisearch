# Codex Review - Update-Server Dashboard Design

## Findings

WARN - `PackageDTO` does not accurately name the served manifest fields. The design says `packages[]` exposes `digest` and a generic `sizeBytes`, and `activeBaseline` exposes `sizeBytes` (`work/11-dashboard/01-update-server-dashboard-design.md:90-95`). The actual remote manifest has `sha256`, `compressed_size_bytes`, and `uncompressed_size_bytes` on both `BaselineRef` and `RemotePackageEntry`, not `digest` or `size_bytes` (`crates/jurisearch-package/src/manifest/remote.rs:55-75`, `:79-99`). If the DTO intentionally normalizes those names, the design still needs to state the mapping explicitly because the review instructions require producer-contract accuracy. Concrete fix: change the DTO to `sha256`, `compressedSizeBytes`, and `uncompressedSizeBytes` (or document exact source-to-DTO aliases) and keep `digest` reserved for catalog/embedded-manifest digest concepts that are not in `RemotePackageEntry`.

WARN - The packages provider section treats `core/manifest.json` as if it were a bare `RemoteManifest`, but the source writes and reads it as `Signed<RemoteManifest>` with the manifest under `payload`. `build_remote_manifest` returns `Signed<RemoteManifest>` (`crates/jurisearch-package-build/src/remote_manifest.rs:63-69`), and producer status parses the served root as `Signed<RemoteManifest>` before reading `signed.payload.head_sequence`, `signed.payload.generated_at`, and `signed.payload.active_baseline` (`crates/jurisearch-producer/src/status.rs:245-264`). Concrete fix: update §4/§5 to say `PackagesProvider` parses the signed wrapper, validates `payload` (and optionally surfaces/verifies the top-level signature), then maps `payload.active_baseline` and `payload.packages`.

WARN - The exit-class model omits the in-flight `running` persisted value and would misbucket a live run if `severityOf(exitClass)` is applied mechanically. `RunRecord::started` persists `outcome = running`, `ended_at = None`, and `exit_class = "running"` (`crates/jurisearch-producer/src/runrecord.rs:70-82`), while `exit_code_for` only knows success classes plus failure buckets and would fall through unknown strings to `70` permanent (`crates/jurisearch-producer/src/exit.rs:40-51`). The design lists `RunRecordDTO.outcome = running` but calls `exitClass` "the EXACT string" and defines `severityOf(exitClass)` without a running special-case (`work/11-dashboard/01-update-server-dashboard-design.md:86-100`). Concrete fix: add `running` to the RunRecord DTO vocabulary and specify that severity is derived from `outcome` first: `running` is neutral/in-progress, success classes are OK/warn, and failure classes use the `exit_code_for` bucket.

WARN - The exit-class list is incomplete for an implementation contract. The design uses an ellipsis after a partial list (`work/11-dashboard/01-update-server-dashboard-design.md:96-100`), but implementation needs the whole persisted vocabulary and bucket map. `ProducerError::class` can persist `ingest-failed`, `enrich-degraded`, `embed-failed`, `provision-failed`, `storage-failed`, `needs-rebaseline`, `alert-hook-failed`, and `io-failed` in addition to the listed examples (`crates/jurisearch-producer/src/error.rs:75-100`); `exit_code_for` separately maps `upstream-unreachable` as transient even though that class is not produced by `ProducerError::class` today (`crates/jurisearch-producer/src/exit.rs:40-51`). Concrete fix: replace the ellipsis with a table of every known class, source (`SUCCESS_CLASSES`, `ProducerError::class`, plus legacy/explicit `upstream-unreachable` if retained), success/failure, and bucket/code.

WARN - The `StatusDTO` section says the shape mirrors the real status output but omits several status fields that are cheap and operationally relevant, especially `generated_at`, `corpus`, `active_baseline_id`, `last_error`, and `fetch_cursors` (`work/11-dashboard/01-update-server-dashboard-design.md:81-85`). The producer emits all of these (`crates/jurisearch-producer/src/status.rs:73-105`, `:180-212`). Omitting fields is acceptable for page-shaped API DTOs, but not for a claimed producer-contract mirror, and `last_error`/`fetch_cursors` directly support the dashboard's errors/freshness ask. Concrete fix: either rename this as a narrowed `StatusSummaryDTO` with an explicit "dropped fields" list, or include the emitted fields in the shared contract and let pages ignore what they do not render.

WARN - The self-executable Bun asset-embedding claim is directionally plausible but currently overconfident. The design states the binary embeds server code and built SPA assets as a packaging fact (`work/11-dashboard/01-update-server-dashboard-design.md:37-42`, `:209-211`), then later leaves the asset embedding mechanism open (`:271-274`). Bun's executable docs support `bun build --compile` and current full-stack executables, but asset inclusion has version-specific mechanics; the design should not treat "embedded SPA assets" as settled before selecting the exact import/static-map strategy and Bun version. Concrete fix: make this a build requirement with a proof task: pin Bun, compile a minimal Vue build with hashed assets, run the compiled binary on the target-like host, and assert deep-linked SPA routes and static asset responses work without a filesystem `dist/` directory.

WARN - `dist.sh` / `deploy.sh` integration is feasible but underspecified relative to the current scripts. Today `UPDATE_SERVER_BINS` contains only `jurisearch-producer` (`dist.sh:71-74`), `BUILD_BINS` is Cargo-only (`dist.sh:217-265`), the update bundle manifest lists one binary (`dist.sh:446-449`), and `deploy.sh` stages/verifies/installs only `/usr/local/bin/jurisearch-producer` (`deploy.sh:75-84`, `:150-170`, `:261-267`, `:321-331`, `:449-507`). The design correctly says this becomes a two-binary bundle, but the implementation plan must account for a non-Cargo Bun build path, a second SHA/version audit, bundle manifest updates, remote staging/install, service lifecycle, and verification. Concrete fix: keep this as design, but add an explicit constraint that the dashboard build must integrate with the existing checksum/version audit model instead of simply appending a binary copy step.

NIT - `TimerDTO` from `systemctl list-timers` is under-specified. The design lists `unit`, `nextRun`, `lastRun`, `active` (`work/11-dashboard/01-update-server-dashboard-design.md:101-103`), but a robust parser should also carry the service unit or group mapping because timers are named `jurisearch-producer-{group}.timer` and service units are `jurisearch-producer-{group}.service` (`crates/jurisearch-producer/src/render.rs:14-24`, `:87-107`). Concrete fix: include `group`, `timerUnit`, `serviceUnit`, `nextRun`, `lastRun`, and `active` so the UI does not infer group names in multiple places.

NIT - The architecture is mostly sound and not obviously over-engineered for a Bun/Vue app, but §5.2 overstates OCP by saying a future PG provider requires "no edits to existing code" (`work/11-dashboard/01-update-server-dashboard-design.md:131-133`). Adding a provider still needs composition-root wiring, route registration, API client endpoint, and a page/panel. Concrete fix: phrase OCP as "no edits to existing providers/parsers" rather than no edits at all.

## Confirmed Good

- `RunRecordDTO` correctly treats duration as derived. The producer persists `started_at` and optional `ended_at` but no duration (`crates/jurisearch-producer/src/runrecord.rs:41-67`).
- `status` is correctly described as on-disk/no-DB/no-network. `build_status` reads run records, fetch cursors, baseline markers, the served manifest, and the lock (`crates/jurisearch-producer/src/status.rs:107-213`).
- Catalog-only package fields are correctly excluded from Phase 1. The served `RemoteManifest` exposes manifest/package compatibility, size, row count, URI, `sha256`, and signing fields, but not catalog `status`, `published_at`, or package change-window fields (`crates/jurisearch-package/src/manifest/remote.rs:18-34`, `:55-100`).
- The journald group requirement is real. Producer units render as `User={user}` / `Group={user}` with no `SupplementaryGroups=systemd-journal` (`crates/jurisearch-producer/src/render.rs:53-68`), so a separate dashboard service needs journal access explicitly.
- The Phase 2 `Bun.sql` seam is reasonable as a deferred provider, with the caveat already noted elsewhere that CT 110 SSL is broken and the dashboard needs a least-privilege view layer before querying producer working tables.
