## Findings

WARN - Phase 1 still promises package columns that the no-DB sources cannot provide. The proposal says the Packages view will show `status`, `change-seq window`, and `published-at` from `manifest.json` + `corpora_dir` (`work/11-dashboard/00-update-server-dashboard-analysis.md:15`, `:34`, `:66`, `:145`, `:193`). The served manifest types do not contain those catalog fields: `RemoteManifest` has `generated_at`, `head_sequence`, `active_baseline`, and `packages` (`crates/jurisearch-package/src/manifest/remote.rs:18`); `BaselineRef` has kind/sequence/size/digest but no package status, change-window, or published timestamp (`crates/jurisearch-package/src/manifest/remote.rs:55`); `RemotePackageEntry` has package id, from/to package sequence, size, row counts, schema/fingerprint, digest/signature, but no package catalog status, change-window, package kind, or published timestamp (`crates/jurisearch-package/src/manifest/remote.rs:79`). Those omitted values are in `package_catalog`: `included_change_seq_high`, `included_change_seq_low`, `status`, and `published_at` (`crates/jurisearch-storage/src/migrations.rs:977`, `:995`, `:998`, `:1025`). The manifest builder itself reads the DB catalog first, then emits only the client-facing subset into the remote manifest (`crates/jurisearch-package-build/src/remote_manifest.rs:93`, `:105`, `:137`). Fix: narrow Phase 1's Packages table to the fields actually available from the served tree: active baseline plus retained incrementals, package id, kind for the active baseline and incrementals inferred from manifest position or embedded manifest, package sequence/from-to sequence, artifact URI, size, row counts, schema/fingerprint, digest/signature, and perhaps artifact `created_at` from embedded manifests if desired. Do not call this `package_catalog.status`, `published_at`, or `included_change_seq_low/high` unless Phase 2 dashboard views are added.

WARN - The journald access requirement is accurate but under-specified for the deploy/systemd work. The proposal says the dashboard should run as `jurisearch` plus `systemd-journal` so `journalctl -u jurisearch-producer-<group>.service` can read logs (`work/11-dashboard/00-update-server-dashboard-analysis.md:53`, `:178`). Existing producer units run with `User=jurisearch` and `Group=jurisearch` (`crates/jurisearch-producer/src/render.rs:53`) and the current deploy script creates the user but does not add any supplementary group for dashboard log access (`deploy.sh:294`). Fix: make the Phase 1 scope explicit that `jurisearch-dashboard.service` must include `SupplementaryGroups=systemd-journal` or `deploy.sh` must add the service user to that group, and add a deploy verification that `journalctl -u jurisearch-producer-legislation.service -n 1` works under the dashboard identity. Without that, the Logs page can be green in design but fail at runtime.
