# 01 — Design: `jurisearch` (French legal search for LLM agents)

Design proposal for a from-scratch, local-first **search engine + CLI** over the French legal corpus, built to be **called by LLM harnesses that can spawn subprocesses**. Derives from `../00-foundation/search.md` (the seven architectural pillars). **Ambition: the best-in-class French juridic search engine for AI agents** — production-grade, not an MVP or demo (Phase 1 = best-in-class **LEGI/statutory** search; full *French juridic* coverage at Phase 2).

**This is design only — no implementation.**

## One-paragraph summary
`jurisearch` ingests official French legal data **from official DILA/LEGI XML (authoritative, day 1)** — LEGI codes, Cour de cassation via Judilibre, administrative justice via the open-data portal — into versioned **canonical records**, then a **Rust** core normalizes, chunks, and indexes them in **embedded Postgres** for **hybrid lexical+semantic retrieval** (BM25 via `pg_search` + dense via `pgvector`, fused with custom RRF; a cross-encoder reranker is **benchmark-gated**). Embeddings come from a configured **OpenAI-compatible endpoint** (default — a hosted API or a local `llama.cpp`), with in-process Rust as an offline option. It exposes a **token-frugal, two-step `search` → `fetch` contract** (plus `cite` for citation verification, `related` for knowledge-graph traversal, `--as-of` for time-travel queries) to LLM agents **over the CLI only** — one-shot `--json` or a warm **JSONL session** — with **complete inline help** (`help agent`, `help schema --json`) as the discovery surface. The differentiator vs. existing French-law tooling (thin API wrappers): **temporal correctness (the core differentiator)**, structure- and zone-aware chunking (official Judilibre offsets, non-sequential reassembly), a source-backed legal knowledge graph (that never auto-asserts *jurisprudence constante*), citation grounding, and an agent-first CLI — exactly the seven pillars in `00-foundation`, built to a best-in-class bar.

## All product decisions are locked — only validation gates remain (see DECISIONS.md)
1. **Backend spike** — *validate, don't reselect*: does embedded Postgres + `pgvector` + `pg_search` clear every §13.3 criterion (packaging + quality)? Fallbacks engage only on a hard failure (defined precedence).
2. **Embedding model** — start from `bge-m3`; pick the final model on **post-fusion legal metrics**, served over the configured endpoint.
3. **Reranker** — benchmark before Phase 1 via a **pluggable local/HTTP provider**; ship if the gain is material within budget, else defer to Phase 3 with a recorded result.

The **phase claim is resolved, not open:** Phase 1 = best-in-class LEGI; full *French juridic* search (statutes + jurisprudence) requires Phase 2.

Settled across the 2026-06-20 reviews: name (`jurisearch`), Rust runtime (Python offline-only), CLI-only + JSONL session (no MCP/HTTP), **embedded-Postgres backend** (`pgvector` + `pg_search`), **endpoint embeddings** (incl. local `llama.cpp`), **official-XML source from day 1**, benchmark-gated reranker (pluggable provider), inline-help contract, AGPL-3.0 (→ `pg_search`).
