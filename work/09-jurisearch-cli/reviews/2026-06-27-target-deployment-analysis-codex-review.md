# Codex review - target deployment analysis

## Findings

### BLOCKER - `dispatch_session_request` is not the one-shot CLI handler

The analysis says the existing `jurisearch serve` seam delegates to `dispatch_session_request`, "the same handler the one-shot CLI uses," making socket results byte-identical to local runs (`work/09-jurisearch-cli/2026-06-27-target-deployment-analysis.md:64-67`). The current code does not route one-shot CLI commands through `dispatch_session_request`: one-shot command dispatch is `dispatch::run`, whose match arms call emitters/payload builders directly (`crates/jurisearch-cli/src/dispatch.rs:15-71`). `dispatch_session_request` is used by the JSONL session loop and `serve_jsonl` (`crates/jurisearch-cli/src/session.rs:88-121`, `crates/jurisearch-cli/src/serve.rs:99-103`). The session wrappers do adapt to the same payload builders used by the one-shot path (`crates/jurisearch-cli/src/session.rs:33-86`), so the seam is real, but the document overstates the exact handler relationship.

Recommendation: rewrite this claim to say `serve` reuses the JSONL session dispatcher, whose per-command wrappers call the same payload builders as the one-shot CLI. Do not claim the one-shot CLI itself calls `dispatch_session_request`; if keeping the byte-identical claim, ground it in shared payload builders/contract tests rather than the dispatcher identity.

### BLOCKER - The dense-fingerprint failure mode is misstated for explicit dense search

The risk section correctly identifies that main chunk search lacks the explicit embedding-fingerprint preflight that zone search has, but it says a mismatch "silently degrades to lexical" (`work/09-jurisearch-cli/2026-06-27-target-deployment-analysis.md:238-240`). That is only accurate for hybrid mode: hybrid still has a lexical CTE while the dense arm filters `chunk_embeddings` by the query embedder fingerprint (`crates/jurisearch-storage/src/retrieval/sql.rs:92-124`). Explicit dense mode has no lexical arm; it applies the same fingerprint filter in `dense_pool` and can return empty/no dense results instead (`crates/jurisearch-storage/src/retrieval/types.rs:226-232`, `crates/jurisearch-storage/src/retrieval/sql.rs:205-224`). The zone path's own comment describes the general failure more accurately: without a fingerprint check, the dense arm can silently match nothing and produce a false readiness/result error shape (`crates/jurisearch-cli/src/retrieval/zone.rs:5-48`).

Recommendation: revise the risk to say that hybrid can silently become lexical-only, while explicit dense can silently return no dense matches/false no-results; the shared point is that main chunk search needs a fail-closed fingerprint compatibility preflight.
