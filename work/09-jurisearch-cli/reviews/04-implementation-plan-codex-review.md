# Review: work/09 implementation plan

## Findings

### BLOCKER: The shared-server writer path is not explicitly sequenced before later phases depend on it.

Phase 2 adds `SharedServerBackend`/writer pools and Phase 6 says the query service can be built and tested against a one-shot `syncd update` (`work/09-jurisearch-cli/04-implementation-plan.md:91`, `:180-182`), but the current updater and apply substrate are still hard-bound to self-managed `ManagedPostgres`: `jurisearch-syncd` unconditionally starts a private durable PG from `--index-dir` (`crates/jurisearch-syncd/src/main.rs:88-92`), `run_catchup` takes `&ManagedPostgres` (`crates/jurisearch-syncd/src/planner.rs:452-456`), the media/incremental apply entry points take `&ManagedPostgres` (`crates/jurisearch-syncd/src/apply.rs:61-65`, `:803-807`), and `activate_generation_with_guard` opens its own connection from `ManagedPostgres` (`crates/jurisearch-storage/src/generations.rs:959-969`). As written, P6 has no proven way to populate a standalone site PG with the writer identity, and P7's "compose existing seams" claim hides a structural refactor.

Concrete fix: add an explicit P2 sub-gate before P6, e.g. "shared-server one-shot writer/update path." Refactor `trust`/`subscribe`/`status`/`update`, `run_catchup`, `apply_baseline`/`apply_rebaseline`/`apply_incremental`, and generation activation to accept a `StorageBackend`/writer handle or connection provider instead of `ManagedPostgres`, while preserving the current self-managed adapter. The gate should prove baseline, rebaseline, and incremental apply against a standalone PG using the writer role, then prove the read role can see the resulting active topology.

### BLOCKER: Writer-owned readiness is scoped to generation activation and misses incremental apply.

Phase 3A says to move readiness production into the activation transaction and stamp readiness after coverage is complete (`work/09-jurisearch-cli/04-implementation-plan.md:113-115`), and Phase 7 assumes that writer readiness/activation already landed (`:211-216`). That covers baseline/rebaseline activation, but not the normal incremental path: `apply_incremental` mutates the active generation, validates postconditions, and advances `corpus_state.sequence` inside its own transaction (`crates/jurisearch-syncd/src/apply.rs:851-1036`) without calling `activate_generation_with_guard`. The current readiness signature includes `corpus:active_generation:sequence` (`crates/jurisearch-storage/src/ingest_accounting/readiness.rs:32-43`), so every successful incremental changes the signature. If only activation writes the stamp, the read-only service sees a missing/stale stamp after an incremental unless query-time recompute/write is kept, violating the P3A invariant that reads do zero writes (`work/09-jurisearch-cli/04-implementation-plan.md:118-123`).

Concrete fix: change P3A from "activation transaction" to "every writer commit that changes the active read topology." Introduce one writer-owned readiness-stamp helper used by both baseline/rebaseline activation and incremental apply. For incrementals, compute projection+dense coverage over the mutated active generation and upsert the stamp in the same transaction before/with `advance_corpus_cursor`; incomplete coverage or stamp failure must abort the transaction and leave the cursor unchanged. Add negative harness tests for an incremental with incomplete coverage, a successful incremental followed by a SELECT-only read with no recompute/write, and stale-stamp detection after sequence advance.

### WARN: The dependency-light base crates are only mechanically enforced at the thin-client phase.

P1 creates `jurisearch-contract`, `jurisearch-transport`, and `jurisearch-render`, and it explicitly calls out the `RetrievalMode`/`GroupBy`/`RetrievalOptions` storage-cycle hazard (`work/09-jurisearch-cli/04-implementation-plan.md:63-69`, `:253-254`). But the only mechanical dependency-cone check is in P8 for the final thin client (`:239-243`). Because `RetrievalMode` currently lives under `jurisearch-storage` and CLI args import it from there (`crates/jurisearch-storage/src/retrieval/types.rs:211`, `crates/jurisearch-cli/src/args.rs:16`), a backwards conversion direction or renderer dependency leak could land in P1 and not be caught until the client extraction.

Concrete fix: add P1 verification that the three base crates compile and have a clean dependency cone: no `jurisearch-storage`, `jurisearch-embed`, `jurisearch-ingest`, `jurisearch-cli`, `postgres`, or model/runtime dependencies. Also make the enum ownership direction explicit: either move the wire enums down to `jurisearch-contract`, or put conversions in storage/query crates so `jurisearch-contract` never depends on storage.

### WARN: The site allowlist tests should deny every non-site command, not just examples.

The compatibility matrix says the site API exposes only `search`, `fetch`, `cite`, `related`, `context`, `compare`, and `status`, and everything else is absent by construction (`work/09-jurisearch-cli/04-implementation-plan.md:45-54`). P6's negative tests mention `setup`/`model`/`eval` (`:198-199`), but the current local dispatcher also handles `help`, `schema`, `exit`, `expand`, `doctor`, `stats`, `inspect`, `versions`, and `diff` (`crates/jurisearch-cli/src/session.rs:121-153`), plus session-excluded one-shot commands. If the implementation accidentally delegates to `dispatch_session_request`, some of these local-only operations can leak through while the example negatives still pass.

Concrete fix: make the P1/P6 allowlist verification table-driven from the compatibility matrix. Assert that every command outside the `Operation` set, including `expand`, diagnostics, help/schema/exit, model/eval, setup, and session-excluded one-shot commands, returns a session `ErrorObject` from the site dispatcher and never reaches the local dispatcher or any `index_dir`-aware payload.

VERDICT: FIXES_REQUIRED
