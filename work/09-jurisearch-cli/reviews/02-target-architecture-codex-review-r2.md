# Target Architecture Review R2

Review scope: `work/09-jurisearch-cli/02-target-architecture.md`, checked against the current source in `/home/pierre/Work/jurisearch`. I treated the r1 review only as history and judged the revised document against the r2 brief's settled decisions.

## Findings

### WARN 1 - The all-corpora endpoint still needs an indexed search-routing invariant

The revised document is internally consistent about the multi-corpus decision: syncd owns the corpus set, one endpoint exposes all active corpora, and the readiness stamp is an aggregate over active corpus/generation/sequence state (`work/09-jurisearch-cli/02-target-architecture.md:188`, `:358`). The current readiness signature supports that shape: it aggregates `corpus || ':' || active_generation || ':' || sequence` ordered by corpus (`crates/jurisearch-storage/src/ingest_accounting/readiness.rs:28`), and `rebuild_server_views` creates `jurisearch_server` union views across all active generation schemas (`crates/jurisearch-storage/src/generations.rs:836`).

The remaining architectural gap is hot search routing. The storage layer explicitly says BM25/IVFFlat retrieval must target qualified physical generation tables because the indexes live there, while stable views are appropriate for non-indexed reads (`crates/jurisearch-storage/src/generations.rs:10`). `execute_read_sql` currently routes the `>1` corpus case to `jurisearch_server` views and leaves multi-corpus hot indexed search as a follow-up (`crates/jurisearch-storage/src/runtime.rs:283`). `hybrid_candidates_json` builds unqualified `chunks` / `chunk_embeddings` SQL and executes it through `execute_read_sql` (`crates/jurisearch-storage/src/retrieval/sql.rs:86`, `crates/jurisearch-storage/src/retrieval/hybrid.rs:5`, `:137`). With more than one active corpus, that preserves logical correctness but not the "reads scale because they target the active generation" property for indexed search.

Recommendation: add a short invariant to §6/§12 or milestone 2: the query service resolves the active corpus set once per request and expands hot search over the active physical generation schemas under the same request snapshot, then fuses/paginates above those corpus arms. Keep `jurisearch_server` stable views for non-indexed reads/status/readiness. Also make the embedding preflight operation-scoped: every corpus included by an "all corpora" dense/hybrid request must match the server fingerprint, or the request fails closed.

### WARN 2 - "Active generation is fully ready" should be tied to apply-time coverage validation, not just INV-6

The revised readiness ownership decision is feasible, and the Q&A correctly established that the query-time write can move to the writer. The document overstates the source grounding when it says an active generation is always fully ready because syncd validates the full index inventory before activation, per work/08 INV-6 (`work/09-jurisearch-cli/02-target-architecture.md:140`, `:348`). The work/08 acceptance row defines INV-6 as index materialisation before cursor advance plus readiness measured from the active generation (`work/08-jurisearch-server/2026-06-27-acceptance-record.md:21`). In source, syncd builds indexes, validates the signed index contract, validates row-count/digest postconditions, and then activates (`crates/jurisearch-syncd/src/apply.rs:221`, `:226`, `:233`, `:257`; `crates/jurisearch-storage/src/generations.rs:1080`, `:1110`, `:1131`). That proves the generation is structurally indexed and converged to the signed package, but it is not the same check as query-readiness completeness.

The existing readiness gate treats "fully ready" as projection coverage complete and embedding coverage complete (`crates/jurisearch-storage/src/ingest_accounting/readiness.rs:219`, `:244`, `:249`, `:271`). Producer-side dense finalize does verify missing embeddings before building IVFFlat (`crates/jurisearch-storage/src/dense.rs:116`, `:146`, `:159`), but the apply-side package contract records row counts/digests and index-build directives, not an explicit query-readiness coverage assertion (`crates/jurisearch-package/src/manifest/embedded.rs:152`, `:161`; `crates/jurisearch-package-build/src/baseline.rs:300`, `:446`). A signed package produced from incomplete dense coverage would converge and index, but the target "no partial-readiness case at a client site" depends on the new apply-time readiness validation/stamp rejecting that state.

Recommendation: adjust §6/§7/§12 to say work/09 extends INV-6 with an apply-time readiness validation: the activation transaction stamps readiness only after projection and dense coverage are complete for the active topology, and incomplete coverage is an apply failure with the cursor unchanged. If the intended invariant is instead "the producer only publishes dense-ready packages," name that as a producer package acceptance requirement too; otherwise the query service must support an explicit partial stamp and reject dense/hybrid requests rather than assuming full readiness.

### NIT 1 - Read-role visibility should be an acceptance criterion in the roadmap

The document now names the right security property: the query service uses a read-only identity that automatically sees each newly activated generation (`work/09-jurisearch-cli/02-target-architecture.md:268`). The current source is still the self-managed/trust-auth model (`crates/jurisearch-storage/src/runtime.rs:170`) and creates generation schemas/tables dynamically (`crates/jurisearch-storage/src/generations.rs:108`, `:128`, `:188`), so this will be a real shared-server acceptance condition rather than a passive configuration detail.

Recommendation: add one sentence to milestone 1 or 2 that activation must leave the read-only identity able to read `corpus_state`, `index_manifest`, stable views, and the newly active generation schemas at commit time. No SQL mechanics are needed in the architecture doc; it just needs to be a named invariant because every future generation is dynamic.

## Grounding Notes
