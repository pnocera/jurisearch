# Re-review: work/09-jurisearch-cli/03-deployment-design.md

The prior r3 findings are resolved in this revision: read-role visibility is now an activation-path postcondition owned by `CorpusWriter::apply` (`03-deployment-design.md:158-167`, `:372-374`), and the raw `command` string is now mapped through a contract-owned `Operation::parse_command` / `as_command` authority before handler lookup (`03-deployment-design.md:91-102`).

I found one remaining design-contract blocker and one warning below.

## Findings

### BLOCKER: Per-operation request `args` still have no shared contract/validation authority

The design says the request/response envelope and operation vocabulary live once in the lowest crate and are the only wire contract shared by the thin client and server (`03-deployment-design.md:81-95`). It also says the thin CLI front-end parses arguments into a `SessionRequest` (`03-deployment-design.md:62-66`), while the site dispatcher receives raw `args: &Value` and the extracted response builder takes "already-validated args" (`03-deployment-design.md:247`, `:264-270`). The missing owner is the schema/defaulting/validation for each operation's `args` payload.

That is load-bearing for the thin/heavy split. Today's source keeps the query request DTOs inside the heavy CLI crate: `SearchRequest`, `FetchRequest`, `CiteRequest`, and the other session request structs are `pub(crate)` in `crates/jurisearch-cli/src/request.rs` (`:15-18`, `:108-142`), and `SearchArgs::into_request` is explicitly the one-shot mapping that must mirror the session serde defaults (`crates/jurisearch-cli/src/request.rs:79-82`). The warm-session path then parses raw `serde_json::Value` through those same CLI-local structs (`crates/jurisearch-cli/src/session.rs:33-66`). A structural thin client cannot depend on `jurisearch-cli`, and the server/query crate should not depend on the thin-client front-end. If the design leaves only `SessionRequest { command, args: Value }` in the shared contract, implementers must either duplicate the JSON `args` shape/defaults in the thin client and site handlers, violating the document's DRY wire-contract claim, or pull heavy CLI request types into the server/thin boundary, violating the dependency cone the design is trying to create. The server also cannot rely on the thin client to have prevalidated `args`, because a JSONL peer can send arbitrary frames.

Concrete fix: add a first-class "query request DTOs + validation/defaults" authority to §3.1 and the DRY ledger, in a dependency-light contract crate such as `jurisearch-contract` or a sibling `jurisearch-query-contract`. It should own typed per-operation request structs and either `Operation::parse_args` or per-operation validators. The thin CLI should map clap args into those DTOs before serialization, and `SiteDispatcher` / `OperationHandler` should deserialize and validate the same DTOs after `Operation::parse_command` and before invoking the response builder. Server-owned fields such as `index_dir` should be absent from the wire DTOs, or stripped by a contract-owned sanitization rule, not handled independently by client and server code. The response builder should then take typed validated request DTOs, not an unspecified "validated args" value.

### WARN: Command parsing is still typed as package/preflight `Reject`

The new operation mapping fixes the previous unattached-vocabulary issue, but its error type is still drawn as `Result<Operation, Reject>` for unknown / legacy / admin / model / ingest / eval commands (`03-deployment-design.md:92-102`). The DRY ledger later points `Reject` at the work/08 §6.3 vocabulary (`03-deployment-design.md:371-375`), and that vocabulary is the package/apply/preflight set: `client_too_old`, `schema_ahead`, `missing_entitlement`, `sequence_gap`, `wrong_generation`, `embedding_fingerprint_mismatch`, `builder_version_mismatch`, `signature_invalid`, `digest_mismatch`, `extension_missing`, `baseline_required` (`work/08-jurisearch-server/2026-06-26-central-ingest-package-distribution-design.md:464-469`; implemented in `crates/jurisearch-package/src/reject.rs:13-40`). None of those codes means "unknown session command" or "admin command not exposed on the site API".

If followed literally, implementers either have to map command/transport failures onto nonsensical package reject codes or extend the package reject enum with session protocol errors, weakening the conformance boundary work/08 just made closed. Today's session surface already has a separate `ErrorObject`/`ErrorCode` shape for bad input and not-implemented commands (`crates/jurisearch-core/src/error.rs:4-14`, `crates/jurisearch-cli/src/session.rs:127-152`), which is the more natural home for command parsing failures.

Concrete fix: distinguish the error authorities explicitly. Keep work/08 `RejectCode` / `RejectError` for package verification, apply preconditions, and embedding-fingerprint preflight. Make `Operation::parse_command` return the shared session/transport error shape instead, for example `Result<Operation, ErrorObject>` or a small `OperationParseError` that converts to `SessionResponse::Err`. Update the DRY ledger so "Reject vocabulary" does not appear to cover command allowlist failures.
