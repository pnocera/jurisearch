# Updated design review: `jurisearch`

Review date: 2026-06-20

Reviewed files:

- `work/01-design/README.md`
- `work/01-design/DESIGN.md`
- `work/01-design/DECISIONS.md`
- `work/01-design/RESEARCH.md`

## Verdict

The updated design is materially better than the previous proposal. It now matches the main product decisions: `jurisearch` as the binary name, Rust for search/runtime, Python only as an offline ingestion helper, CLI-only with JSON/JSONL, no MCP/HTTP, Qdrant out of scope, and embedded Postgres + `pgvector` + `pg_search` as the preferred backend spike.

I would not restart the design. The next pass should tighten a few contracts that are currently underspecified or slightly inconsistent.

## Findings

### P1: The `AgentPublic/legi` bootstrap path must not import its chunks as final chunks

`DESIGN.md` correctly treats `AgentPublic/legi` as an optional bootstrap, not the source of truth. The remaining risk is that the implementation could ingest the HF rows directly as retrieval chunks because they already contain `chunk_text` and `embeddings_bge-m3`.

That would violate the design's own structure-aware chunking rule. The dataset card says long articles are split with LangChain `RecursiveCharacterTextSplitter`, `chunk_size=1024`, `chunk_overlap=0`, and BGE-M3 embeddings are stored as stringified vectors. That is useful for a quick demo, but it is not the final legal chunking model.

Required clarification:

- `AgentPublic/legi` rows may seed canonical article-version records.
- Rust ingest must reconstitute article/version-level records by `doc_id` before final indexing.
- Final chunks and embeddings should be generated by `jurisearch-ingest`, using the project chunking rules and model manifest.
- Precomputed HF embeddings may be used only for a fast smoke test, not for the authoritative index.

Relevant design locations: `DESIGN.md` §4, §6, §13.4, §17.

### P1: Judilibre `rapprochements` should be modeled as relationships, not text zones

The data model currently lists `rapprochements` inside `zones` with character offsets. Judilibre's public spec describes zones as text delimitations such as introduction, exposé, moyens, motivations, dispositif, and annexes; it separately lists texts applied and jurisprudence rapprochements as decision metadata.

The design already says `rapprochements` feed the graph and are not chunked for retrieval, which is right. The schema should make that unambiguous:

- `zones`: only text-offset ranges.
- `related_decisions` or graph edges: publisher-provided `rapprochements`.
- `texts_applied` or graph edges: publisher-provided applied texts.
- `edge_source`: `publisher` or `inferred`.

Also store raw Judilibre taxonomy keys alongside normalized labels. Publication levels can arrive as arrays of short keys such as `b`, `r`, `l`, `c`, while the design examples use human labels like `B`, `P`, and `Inédit`.

Relevant design locations: `DESIGN.md` §5.1, §6, §8.

### P1: Citation verification needs a dedicated contract

The design has a `jurisearch cite` command and the right principle, but the verification behavior is not specified deeply enough. There is also a stale cross-reference in the pillar table that points citation verification to "§11 Grounding", while §11 is now the CLI latency section.

Add a dedicated "Grounding and citation verification" section, or expand §10. It should define:

- Accepted inputs: internal IDs, LEGIARTI/LEGITEXT/LEGISCTA, ECLI, pourvoi, NOR, CETATEXT, and free-text citations.
- Match result states: exact, normalized, ambiguous, stale-version, not-found, source-unavailable.
- Whether verification checks only the local index or can call official APIs when online.
- What `--strict` changes.
- The JSON response schema for successful and failed verification.
- How date-sensitive citations interact with `--as-of`.

Relevant design locations: `DESIGN.md` lines around the pillar table, §10.1, §10.2, §16.

### P1: JSONL session mode needs an explicit wire protocol

The design correctly adds `jurisearch session --jsonl` to avoid repeated model/index startup. The remaining issue is that the protocol is described only in prose.

Define the wire envelope now, before implementation:

```json
{"id":"req-1","command":"search","args":{"query":"article 1240","kind":"code","top_k":5}}
```

and response shape:

```json
{"id":"req-1","ok":true,"result":{...}}
{"id":"req-2","ok":false,"error":{"code":"bad_input","message":"...","suggestions":[]}}
```

The contract should also say:

- Responses preserve input order unless future parallelism explicitly opts out.
- `stdout` is JSONL only; diagnostics go to `stderr`.
- `help` and `help schema` work inside a session.
- `exit` has a response or clearly documented no-response behavior.
- Malformed JSON lines return JSONL errors rather than breaking the process, unless a fatal mode is chosen.

Relevant design locations: `DESIGN.md` §10.1, §10.4, §11, §15.

### P1: Embedded Postgres packaging is the right spike, but the artifact contract is incomplete

The preferred backend is now sensible: embedded Postgres, `pgvector`, and `pg_search`. Current docs support the direction: `pg_search` is a BM25 Postgres extension built on Tantivy; `pgvector` stores vectors in Postgres; `pg-embed` and `postgresql_embedded` can run local Postgres from Rust.

The design should state explicitly that "embedded Postgres" still means a managed local Postgres process, not an in-process SQLite-like library. That affects packaging and operations.

Add acceptance criteria for the backend spike:

- Bundled vs downloaded Postgres binaries.
- Bundled extension versions for `pgvector` and `pg_search`.
- Extension install/migration flow.
- Unix socket or ephemeral-port binding, with no public network exposure by default.
- Single-writer locking across simultaneous `jurisearch` processes.
- Crash recovery and clean shutdown.
- Index upgrade/migration story.
- Cross-platform target policy, even if v1 is Linux-only.

Relevant design locations: `DESIGN.md` §13.1, §13.3, §13.5.

### P1: "No network at query time" needs a model-cache rule

The design says no network at query time, but `fastembed-rs` downloads models on first use and loads from cache afterwards. That means a first `search` could silently touch the network unless the model has already been installed.

Add one of these requirements:

- `jurisearch model fetch bge-m3` or `jurisearch setup models` prepares local models before query use.
- `jurisearch status` reports missing local models.
- `jurisearch search` fails with an actionable error if local mode is configured but the model is not cached, unless an explicit `--allow-download` flag is passed.

This matters for confidential legal work and reproducible offline use.

Relevant design locations: `DESIGN.md` §2, §11, §13.1, §14.

### P2: `kind` naming is inconsistent between CLI and schema

The CLI uses `--kind code|decision|all`, while the document schema says `kind` is `"article" | "decision"` and search output examples return `"kind": "article"`.

This is small, but agents will see both. Pick one public vocabulary:

- CLI-facing: `code`, `decision`, `all`.
- Schema-facing: maybe `kind: "article"` internally, but expose a separate public `source_kind` or document the mapping.

The JSON schema from `jurisearch help schema --json` should be the final arbiter.

Relevant design locations: `DESIGN.md` §5.1, §10.1, §10.2.

### P2: Temporal normalization needs sentinel handling

The design uses `valid_to: null` for in-force articles, while `AgentPublic/legi` examples use `end_date = 2999-01-01`. Official source data and derived corpora may use sentinels or missing dates differently.

Specify canonical normalization:

- Convert open-ended validity to `valid_to: null`.
- Preserve the raw source value in detailed/provenance metadata.
- Define interval semantics for `null`: `valid_from <= as_of && (valid_to is null || as_of < valid_to)`.
- Include edge tests around reform dates and same-day version changes.

Relevant design locations: `DESIGN.md` §5.1, §10.2, §12.

### P2: `help schema --json` should work without an index

The design says schemas are shipped in the index `schemas/` directory and served by `help schema --json`. Because inline help is the API discovery surface, schema help should work before any corpus is installed.

Recommendation:

- Compile command/request/response/error schemas into the binary.
- Copy the same schema files into the index artifact for provenance.
- `help schema --json` reads from the binary by default, and optionally reports the active index schema version when an index is available.

Relevant design locations: `DESIGN.md` §10.4, §13.5, §15.

## What Improved

- The design now correctly commits to `jurisearch` instead of `juri`.
- Runtime/search is Rust-first, with Python limited to offline canonical-record generation.
- CLI-only is now coherent: one-shot JSON plus JSONL session mode.
- MCP/HTTP/`serve` and Qdrant are out of v1.
- The backend spike is now pragmatic: embedded Postgres + `pgvector` + `pg_search`, with FTS/Tantivy fallbacks.
- Reranking is staged instead of overpromised.
- Inline help is correctly elevated to part of the API.
- Evaluation now includes CLI/JSON/session behavior, not only retrieval metrics.

## Research Notes

- `AgentPublic/legi` has 629k rows, Etalab licensing, LEGI-like metadata, `chunk_text`, and BGE-M3 embeddings, but its dataset card says long articles are chunked with `RecursiveCharacterTextSplitter` at `chunk_size=1024` and embeddings are stringified vectors: https://huggingface.co/datasets/AgentPublic/legi
- Judilibre's public OpenAPI spec exposes text zones as offsets, and separately describes applied texts and jurisprudence rapprochements: https://raw.githubusercontent.com/Cour-de-cassation/judilibre-search/dev/public/JUDILIBRE-public.json
- `pg_search` is a Postgres extension for BM25 full-text/faceted/hybrid search, built on Tantivy and licensed AGPL-3.0: https://pgxn.org/dist/pg_search/
- `pgvector` supports exact and approximate vector search inside Postgres, with multiple vector types and distance metrics: https://github.com/pgvector/pgvector
- `pg-embed` can run local Postgres from Rust and downloads/caches PostgreSQL binaries: https://docs.rs/pg-embed
- `postgresql_embedded` provides an embedded-like Rust API, but Postgres still runs in a separate process; it can bundle or download Postgres and supports extensions: https://github.com/theseus-rs/postgresql-embedded
- `fastembed-rs` supports BGE-M3 and rerankers, but models download on first use and then load from cache: https://github.com/anush008/fastembed-rs

## Bottom Line

The updated design is directionally ready. I would address the P1 items before implementation, especially the `AgentPublic/legi` bootstrap semantics, Judilibre zone-vs-relationship modeling, citation verification contract, JSONL wire protocol, and embedded Postgres packaging rules.
