# Option A vs Option B Decision

Recommendation: build **Option B**, a separate official-zone retrieval subsystem, not zone units inside the existing main `chunks` / `chunk_embeddings` retrieval path.

The deciding reasons are:

1. **Option B preserves the proven whole-decision retrieval path.** The current hybrid search path is tightly centered on `HybridCandidateQuery` over `chunks`, `chunk_embeddings`, and `documents`, with filters limited to `kind_filter` plus `DecisionFilters` (`crates/jurisearch-storage/src/retrieval.rs:71-86`, `:269-385`, `:556-724`). A1 would replace Cassation body retrieval text, and A2 would add duplicate official zone units into that same ranking pool. Either way, the default topical retrieval path changes, including RRF input pools, document grouping, snippets, cursor ordering, dense coverage, and probes behavior. Option B lets the existing path stay the default path for non-zone search.

2. **The Phase 2 honesty model is cleaner as a separate subsystem.** Bulk jurisprudence records are validated as `chunking_provenance = "heuristic"` (`crates/jurisearch-ingest/src/juri/mod.rs:190-196`), generated as `decision_summary` / `decision_body` heuristic chunks (`crates/jurisearch-ingest/src/juri/mod.rs:754-823`), and gated by the Phase 2 check that every bulk jurisprudence source reports `zone_accurate=false` (`crates/jurisearch-cli/src/main.rs:8800-8824`). Official Judilibre zones are already modeled as a per-decision overlay in `decision_zones`, explicitly kept separate from canonical bulk records to preserve that honesty boundary (`crates/jurisearch-storage/src/migrations.rs:438-470`). Option B extends the same boundary into retrieval instead of weakening the meaning of the main corpus chunks.

3. **A1 is the wrong tradeoff for snippets and `fetch`.** Search snippets come from `chunks.body`, while `fetch_documents_json` returns the canonical `documents.body` plus all persisted chunks (`crates/jurisearch-storage/src/retrieval.rs:269-385`, `:757-850`). Replacing Cassation `decision_body` chunks with Judilibre-derived zone text would make default search and fetch speak different serializations of the same decision. That mismatch is not a small UI issue; it would make retrieved snippets harder to reconcile with the fetched decision body.

4. **A2 avoids A1's replacement risk, but imports most of B's complexity into the main index.** To make A2 honest, the schema would need a first-class retrieval-unit type, zone label, official/heuristic provenance, and a default-query exclusion rule so non-zone searches do not double-count decisions through both heuristic body chunks and official zone chunks. Today the dense rebuild input loader scans all rows in `chunks` for missing/stale embeddings (`crates/jurisearch-storage/src/dense.rs:49-75`), and the candidate SQL has no concept of "zone-only units" to exclude. That means A2 would touch the common embedding/readiness/retrieval path anyway, but without the clean operational boundary B gives.

5. **The partiality is structural, so the product should expose a distinct capability.** Official-zone retrieval currently means resolver-reachable Cour de cassation decisions only, about 494,701 decisions in the analyzed index, not `capp`, not `jade`, and not Cassation records without a parser-valid pourvoi. A separate zone subsystem can report "official Cassation-zone search" as its own coverage-bounded retrieval mode. If official zone units live beside normal chunks in the main index, it becomes easier for callers to mistake a partial official overlay for corpus-wide zone precision.

The main risk I am accepting is the operating cost of a second retrieval index: separate BM25/dense structures, embedding coverage, refresh/backfill accounting, evals, and explicit routing for `--zone motivations` / `moyens` / `dispositif`. That is real cost, but it is the right cost to pay because it localizes the new official-zone semantics instead of making every existing decision query prove it has not been contaminated by zone-only units.

I would switch only under a stronger requirement than the one here: if jurisearch deliberately wants one unified ranking surface that blends whole-decision and zone-local signals for all decision queries, and the schema first grows explicit retrieval-unit type, zone, provenance, default-exclusion, fetch serialization, readiness, and eval contracts. Under that condition, the fallback would be **A2**, not A1. I would not choose A1 unless `fetch` also moved to the same Judilibre serialization for the affected decisions, which would be a much larger corpus-contract change.
