# Phase 1 Code Review

## Findings

1. **P1 - `eval run` qrel metrics can overstate quality by ignoring relevant qrels outside the retrieved pool.** In `eval_run_payload`, qrels are loaded into `result.labels` from the full qrels file (`crates/jurisearch-cli/src/main.rs:1266`), but `compute_eval_metric` receives `pool_uids` built only from retrieved candidates (`crates/jurisearch-cli/src/main.rs:1336`) and then derives the relevant set for recall and ideal list for nDCG from that pool only (`crates/jurisearch-cli/src/main.rs:1038`, `crates/jurisearch-cli/src/main.rs:1059`). If a relevant qrel document is not returned by any mode, it disappears from the denominator and from IDCG, so recall/nDCG can look perfect even when every retriever missed judged-relevant documents. Fix: separate the judged relevance universe from the retrieved pool. For qrels, build the relevance universe from all qrels for the question; for external judges, use the judged pool. Pass that universe into recall and nDCG, and keep the existing `None` behavior only when that universe has no label >= `rel_min`.

2. **P1 - `eval run --modes bm25` still requires the embedding runtime and performs embeddings.** `eval_run_payload` unconditionally prepares `PreparedQueryEmbedder::from_env()` (`crates/jurisearch-cli/src/main.rs:1193`) and unconditionally embeds every question (`crates/jurisearch-cli/src/main.rs:1206`) before checking whether any requested mode uses dense retrieval. A lexical-only qrels eval should be able to run with BM25/index readiness only, but this path will fail if the embedding endpoint or model cache is unavailable. Fix: construct the embedder only when `modes.iter().any(RetrievalMode::uses_dense)`, call `ensure_query_readiness` with `SearchLexical` for BM25-only runs, and only call `embed()` inside the dense/hybrid branch.

3. **P1 - Precision@k uses the returned-list length instead of k.** In `compute_eval_metric`, precision is calculated as `hits / topk.len().max(1)` (`crates/jurisearch-cli/src/main.rs:1040`). Standard P@k divides by `k`, treating missing ranks as non-relevant; otherwise a short page with one relevant document yields `p@10 = 1.0` instead of `0.1`. This is especially relevant because document grouping can exhaust the pool before k. Fix: divide by `spec.k` for precision@k, or explicitly rename/document a nonstandard precision-over-returned metric and do not expose it as `p@k`.

4. **P1 - `doctor` does not implement several required T1.4a preflight checks.** The plan calls for doctor to aggregate PG data-dir state, migrations-current status, model-fingerprint compatibility, model cache, and index/replay readiness without owning Postgres. The implementation deliberately stops at config, endpoint, model cache, extension files, and index-dir existence, and says to run `status` for migrations/index readiness (`crates/jurisearch-cli/src/main.rs:5406`, `crates/jurisearch-cli/src/main.rs:5480`, `crates/jurisearch-cli/src/main.rs:5504`). That leaves T1.4a materially incomplete: a doctor response can be "ready" without proving the configured index schema/replay state is usable. Fix: add non-owning index checks that do not call the owning `open_index` path: report whether an index data dir exists, whether a server for it is reachable/attachable if expected, schema migration version vs `CURRENT_SCHEMA_VERSION`, stored embedding fingerprint compatibility, and replay/query-readiness state; if a non-owning check cannot be performed, return an explicit warn/fail check rather than omitting it.

5. **P2 - The command contract/help is still drifting for Phase 1 surfaces.** `SearchArgs` exposes `--group-by` (`crates/jurisearch-cli/src/main.rs:219`), but `SearchRequest` in `compiled_schema()` does not include a `group_by` property (`crates/jurisearch-core/src/schema.rs:78`). Agents reading `help schema --json` cannot discover the new document grouping request field, despite it being the centerpiece of T1.2a and accepted in session JSON. Separately, the top-level clap help for `related` still says `STUB - not yet implemented` (`crates/jurisearch-cli/src/main.rs:151`) while the contract marks it implemented (`crates/jurisearch-core/src/contract.rs:96`). Fix: add `group_by` with enum/default to `SearchRequest`, update tests to assert it, and replace the stale `related` clap text with the implemented depth-1 relation summary and example.
