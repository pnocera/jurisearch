# Answer

(B) is acceptable for P0, and I would prefer it over forcing an explicit `Corpus` through the CLI for the citation-resolution paths. The clean contract is: storage writers are responsible for deriving `corpus` from authoritative relational ownership, and the schema makes failure impossible to ignore with `corpus text NOT NULL` plus SQL that returns exactly one corpus or errors. For `official_api_responses`, derive from `subject_document_id -> documents.corpus` when present; otherwise derive from `citation_key -> decision_legislation_citations -> documents.corpus`, requiring `count(distinct corpus) = 1`. For `legislation_citation_resolutions`, derive the same way from its occurrences, since `citation_key` is deliberately deduped and has no single owning decision. I would not use a broad single-corpus fallback in normal writers because it can silently turn an unlinked row into a replicated row once the database is mostly `core`; keep any fallback limited to a migration/backfill helper or a clearly named bootstrap-only path. If a Legifrance archive/resolution has no occurrence yet, or spans corpora, fail loudly and let the caller/order of operations be fixed. A hybrid is only useful for ergonomics: Judilibre call sites may pass/derive through the subject document naturally, but the invariant should still be enforced centrally in storage, not trusted to CLI threading.
