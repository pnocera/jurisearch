# Answer

Keep the producer on the existing `public` authoritative working set for P2. Moving both producer and client into `jurisearch_server_<corpus>_gNNNN` now would be a wide rewrite of ingestion, projection, outbox rematerialisation, and package-build assumptions for little immediate gain. The producer's job is to maintain the mutable canonical corpus plus `package_change_log`; the client's job is to apply immutable-ish package generations and switch them. Those are different lifecycles. The pitfall is not "producer stays public"; the pitfall is using one connection mode for both roles. Add explicit connection/read roles: producer/write connections keep default `public`; package apply writes only with schema-qualified physical generation names; client compatibility reads may use `search_path`; hot indexed retrieval should get an explicit active-generation target.

Do not bet P2 correctness/performance on `search_path = jurisearch_server, public` making the current BM25/IVFFlat SQL transparently work through `UNION ALL` views. Plain PostgreSQL can often push predicates through simple views, but the current hot paths are not plain: they use `paradedb.score(c.chunk_id)`, `@@@`, vector `<->`, `ORDER BY distance LIMIT`, and joins across `chunks`/`chunk_embeddings`/`documents` and `zone_units`/`zone_unit_embeddings`. The BM25 and IVFFlat indexes live on the physical per-generation tables; the stable views themselves are not indexed. For non-indexed reads, stats, fetch, and compatibility surfaces, stable views are fine. For search/zone search, introduce a small resolver that reads `jurisearch_control.corpus_state` and returns the active physical schema(s), then have the SQL builder emit qualified physical table names for the relevant corpus set. For multi-corpus search, build per-corpus indexed arms against each active physical schema and `UNION ALL` the ranked candidates above that. Do not use a `search_path` containing active generation schemas for this: it cannot represent multiple active corpora for the same relation name, and it becomes connection-state you would have to refresh after every switch.

Per-connection `SET search_path` is acceptable only as a compatibility/read convenience, and only because the switch is still view replacement. The design warning was correctly against `ALTER DATABASE SET search_path` as the switch mechanism. Set it on every client read connection, not globally, and keep it pointing at the stable namespace, not at generation schemas. I would avoid relying on `public` fallback for replicated relation names: either ensure the client has no public replicated tables, or add tests that every replicated relation has a `jurisearch_server` view so a missing view cannot silently hit stale public tables. If `public` is needed for extension types/functions such as `vector`, keep that as an extension-resolution detail, not as a data fallback. Never set this read search path on producer ingest connections or generation-apply writer connections.

For the P2 primitives, the shape is right, but make the control schema stricter. `corpus_state` should be the only activation authority and should include at least corpus, active_generation, sequence, baseline_id, schema_version, embedding_fingerprint, builder_versions, last package id/digest, and applied_at. `generation_registry` should track corpus, generation, physical schema name, state (`building`, `active`, `retired`, plus probably `failed`), created_at, activated_at, retired_at, source package/baseline id, and a validation/postcondition digest. Enforce one active generation per corpus with a partial unique index. The switch transaction should take the apply advisory lock, validate the registry row is `building`, validate postconditions/index readiness, replace the stable views for that corpus, update `corpus_state`, mark the old generation retired and the new one active, then commit. Retire/drop should be async with lock timeouts; never drop the old generation in the switch transaction.

One implementation warning: "reusing migration DDL" should not mean replaying the current global migrations verbatim inside every generation schema. Split the DDL into reusable fragments: extension/control/global DDL stays global; replicated physical table/index DDL can be emitted qualified into a generation schema; operational `ingest_*`, `package_change_log`, `schema_migrations`, and `index_manifest` handling need explicit classification. This classification is the important P2 boundary, and it should be tested before P3 depends on it.
