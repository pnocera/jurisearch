# Answer

Yes: thread an `OutboxContext` into the storage writers. That is the right seam because the writer already owns the semantic mutation and the transaction, while the CLI only knows command intent. I would keep it as a small typed context rather than loose `run_id`/`schema_version` params, and make it mandatory for replicated mutations once P1 is enabled. Do not emit at a higher layer: it will miss rollback coupling and will not know whether the writer performed an upsert, delete, or set replacement. Keep corpus derivation inside storage from authoritative links/columns, as in P0; the CLI should not become the corpus oracle.

The proposed mapping is mostly right, with two adjustments. First, per-document scope is correct for `chunk_embeddings`; the builder can materialize all embeddings for the touched document, and per-chunk outbox rows would only add noise. But when a document projection can change chunk membership, partitioning, contextualized body, or graph-edge membership, do not model that as a plain `upsert` of "doc + chunks + graph edges" unless the writer also deletes stale child rows. Current projection upserts current chunks/edges, so the safe event is a document-scoped `replace_set` for the affected child set, or a paired document `upsert` plus `chunks_with_embeddings`/edges replacement scope. Second, `decision_zones` removal and hierarchy-backfill embedding invalidation should be `replace_set` when the meaning is "the complete derived set for this document is now empty/current"; reserve `delete` for rare base-row removals/redactions. For hierarchy backfill, a document-scoped `replace_set` for `chunk_embeddings` is clearer than `delete` because the current embedding set after the operation is deliberately empty/stale-pending, not a base deletion.

Hook coverage should include every §4.2 replicated writer: `documents`, `chunks`, `chunk_embeddings`, `graph_edges`, `legi_metadata_roots`, `decision_zones`, `zone_units`, `zone_unit_embeddings`, `official_api_responses`, `decision_legislation_citations`, and `legislation_citation_resolutions`. Your list covers those, with the caveat above that document projection owns child chunk/edge stale-row semantics. `index_manifest`, `schema_migrations`, and `ingest_*` stay intentionally hookless.

Keep `ingest_run_id` NOT NULL, but treat it as a producer run id rather than only a LEGI/juri archive ingest id. Derived rebuilds, embedding jobs, zone enrichment, legislation collection/enrichment, and hierarchy backfill should each carry their own run id created at command start. If the column name is still changeable, `producer_run_id` or `operation_run_id` is more honest; if the design name is fixed, keep `ingest_run_id` and define it as "the run that produced this mutation." Nullable would weaken auditability and make package-window debugging worse.
